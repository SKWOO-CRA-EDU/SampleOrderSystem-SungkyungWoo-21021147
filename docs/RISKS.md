# RISKS

> 본 문서는 `docs/DECISIONS.md`(Q1~Q16 채택 결정)와 `docs/SPEC_ANALYSIS.md`(SPEC 원문 분석)를 재검토하여,
> 생산관리 시스템 실무 경험 관점에서 향후 구현·운영 단계에서 실제로 문제가 될 가능성이 높은 리스크를 우선순위(위험도) 순으로 정리한다.
> 각 항목은 "무엇이 문제인가 / 왜 실무에서 터지는가 / 어떻게 완화할 것인가 / 업계 통상 채택 방식(웹 리서치)"으로 구성한다.
> "업계 통상 채택 방식"은 각 항목에 대해 별도로 웹 검색을 수행하여, 실제 MES/ERP/일반 소프트웨어 업계에서 통용되는 구체적 기법과 근거 출처를 정리한 것이다.

## 1. (Critical) 단일 프로세스 전제(Q10)가 실제 운영 환경과 어긋날 가능성

**문제**: Q10은 "멀티 프로세스/멀티유저 동시접근은 범위 밖"으로 결정했으나, 실제 현장에서는 주문담당자·생산담당자가 서로 다른 PC/터미널에서 동시에 프로그램을 띄우는 경우가 흔하다. 학생 프로젝트 채점 시나리오에서도 "두 창을 띄워 테스트"하는 방식으로 이 known limitation이 쉽게 재현되어 감점 포인트가 될 수 있다.

**왜 실무에서 터지는가**: "단일 사용자 전제"는 문서로만 존재하고 프로그램 자체는 이를 강제하지 않는다(파일 락, 프로세스 잠금 없음). 사용자는 문서를 읽지 않고 프로그램을 두 번 실행해볼 수 있으며, 이 경우 같은 저장 파일에 두 프로세스가 동시에 쓰기를 시도해 데이터가 깨지거나(마지막에 저장한 프로세스가 이전 저장 내용을 덮어씀) 재고 오버부킹이 실제로 재현된다.

**완화**: 최소한 "이미 실행 중인 인스턴스가 있으면 경고 후 종료" 수준의 단일 인스턴스 가드(예: 파일 기반 lock 파일 존재 여부 체크)를 넣는 것을 고려. 최소 비용으로 Q10의 known limitation을 "문서상 제약"에서 "코드로 강제되는 제약"으로 격상시켜야 실제 사고를 막을 수 있다.

**업계 통상 채택 방식 (웹 리서치)**:
- **OS 수준 Named Mutex(Windows, 가장 흔한 방식)**: `System.Threading.Mutex`를 GUID가 포함된 전역 이름(`Global\MyApp-{GUID}`)으로 생성하고, 두 번째 실행 시 `createdNew == false`(이미 존재)를 감지해 즉시 종료(또는 기존 창 활성화). 프로세스가 비정상 종료되면 OS가 자동으로 뮤텍스를 해제하므로 별도의 "stale lock 정리" 로직이 필요 없다는 것이 가장 큰 장점.
- **배타적 파일 잠금**: `new FileStream(path, FileMode.Open, FileAccess.ReadWrite, FileShare.None)`으로 저장 파일 자체를 앱 생애주기 동안 배타 오픈 상태로 유지. 두 번째 프로세스가 같은 경로를 열려고 하면 즉시 `IOException` 발생. 이 역시 프로세스 종료 시 OS가 핸들을 닫아주므로 자연 해제됨.
- **PID 기반 사이드카 lock 파일(`app.lock`)**: 실행 시 lock 파일의 PID를 읽어 해당 프로세스가 살아있는지 확인(staleness 검사) 후 진행. 크래시 시 orphan lock 파일이 남는 문제가 있어 위 두 방식보다 구현이 번거로움.
- **채택 권고**: 이 프로젝트(.NET 콘솔 앱, 단일 데이터 파일)에는 Named Mutex 또는 파일 배타 잠금(`FileShare.None`) 중 하나가 가장 적합 — 둘 다 크래시 시 자동 해제되어 별도 정리 로직이 불필요.
- 출처: [Single-Instance Mutex Guide](https://comcomponent.com/en/blog/single-instance-mutex-guide/), [Single-Instance .NET Apps: Mutexes, Named Pipes, UX](https://www.dotnet-guide.com/how-to-restrict-a-program-to-single-instance-in-net.html), [pid — Pidfile with stale detection (GitHub)](https://github.com/trbs/pid)

## 2. (Critical) 재고 차감/증가 로직의 원자성(atomicity) 부재로 인한 정합성 붕괴

**문제**: Q9(승인 시 즉시 차감), Q7(취소 시 복구), Q13(생산완료 시 실생산량×수율만큼 증가) 세 로직이 모두 "재고수량"이라는 동일한 공유 상태를 수정한다. 각 전이가 트랜잭션처럼 원자적으로 처리되지 않으면(예: 재고 차감 후 상태 전이 저장에 실패) 재고수량과 주문 상태가 서로 어긋난 채로 파일에 영속화될 수 있다.

**왜 실무에서 터지는가**: 실제 MES/ERP 사고 사례의 상당수가 "재고 차감은 됐는데 주문 상태 전이는 실패"처럼 두 단계 쓰기 중 하나만 성공하는 부분 실패(partial failure)에서 발생한다. 특히 파일 기반 영속성(§7)에서 저장 도중 예외가 나면 이런 반쪽짜리 상태가 그대로 디스크에 남을 위험이 크다.

**완화**: 상태 전이 + 재고 증감을 하나의 논리적 단위(단일 메서드/단일 저장 호출)로 묶고, 저장 실패 시 메모리 상의 변경도 롤백하는 원칙을 설계 단계에서 못박아야 한다. 단위 테스트에 "전이 도중 예외 발생 시 재고수량 불변" 케이스를 반드시 포함.

**업계 통상 채택 방식 (웹 리서치)**:
- **단일 원자적 메서드 + 단일 원자적 파일쓰기(이 프로젝트에 가장 적합)**: 상태 전이와 재고 증감을 한 번의 인메모리 연산(예: `ReserveStock(order)`가 재고 확인·차감과 상태 전이를 함께 수행)으로 묶고, 이후 전체 상태를 파일에 "임시파일 작성 후 rename"(NTFS/POSIX 모두 동일 볼륨 내 rename은 원자적)으로 한 번에 반영. 진짜 DB 없이도 ACID 커밋과 유사한 효과를 얻는 방식이며, 다만 "차감 전에 검증부터 마쳐야" 한다(먼저 검증 → 인메모리 변경 → flush 순서 고정).
- **Command 패턴 + 인메모리 롤백**: 각 전이를 before/after 상태를 가진 커맨드 객체로 캡슐화하고, 실패 시 이미 실행된 커맨드를 역순으로 되돌림. 여러 단계가 얽힌 트랜잭션에 유용하나, 이 프로젝트처럼 필드 2개(상태·재고)만 얽힌 경우엔 과한 설계.
- **Write-Ahead Log(WAL)**: 의도(intent)를 먼저 기록 → 적용 → 커밋 표시. 재시작 시 미완료 항목을 재생/폐기하여 크래시 복구를 보장하는 실제 MES/ERP 및 분산 시스템(Saga, 이벤트소싱)의 토대가 되는 패턴이나, 이 규모의 콘솔 앱에는 과설계(overkill).
- **채택 권고**: 단일 원자적 메서드 방식 채택.

## 3. (High) Q13의 수율 이중 적용 해석이 SPEC 원문과 정면으로 충돌할 위험

**문제**: DECISIONS Q13은 "실생산량 = 생산 시도 수, 재고엔 실생산량×수율(정상품)만 반영"으로 결정했다. 그러나 SPEC_ANALYSIS 3.2/3.3에는 "총생산시간 = 평균생산시간 × 실생산량"이 반복 명시되어 있고, §4 검증 절의 화면표기값(206, 165 — Q2에서 이미 "SPEC 오기"로 기각됨) 조차 실생산량을 그대로 쓰는 뉘앙스다. 만약 채점자/발주자가 "실생산량 = 재고에 그대로 더해지는 수량"으로 SPEC을 이해하고 있다면, Q13 해석은 SPEC 재현성 시험에서 정답과 어긋난 결과를 낼 수 있다.

**왜 실무에서 터지는가**: 이런 "역산 공식의 의미 해석" 이슈는 로직 자체는 정교하지만, 검수자가 다른 멘탈모델(실생산량=완제품 수량)로 채점표를 만들었을 경우 구현이 아무리 정확해도 "틀린 결과"로 판정된다. 실무에서도 "공식의 의도"를 팀 내부에서만 합의하고 발주처와 재확인하지 않아 인수 단계에서 뒤집히는 경우가 매우 흔하다.

**완화**: 이 결정은 다른 Q들과 달리 화면에 노출되는 최종 수치(재고수량)에 직접 영향을 주므로, 가능하다면 SPEC 원저자/채점 기준에 재확인이 필요한 항목으로 별도 표시하고, 화면에는 "생산시도수량"과 "정상품수량(재고반영분)"을 함께 노출해 어느 쪽으로 검증하든 답을 확인할 수 있게 방어적으로 설계할 것을 권장.

**업계 통상 채택 방식 (웹 리서치)**: Q13 결정(실생산량×수율만 재고 반영)은 실제 업계 표준과 일치함이 확인됨.
- **SAP PP/MRP의 scrap factor 로직**: SAP는 부족분(net requirement)을 스크랩률만큼 부풀린 "계획 오더 시작 수량"으로 산출한다(예: 정상품 200개가 필요하고 스크랩률 10%이면 220개를 투입 계획). 이때 가용재고/순소요량 계산은 항상 "정상품 기준"으로 이루어지며, 스크랩분은 결코 가용재고로 집계되지 않는다. (SAP Learning, "Calculating Scrap"; SAP Community)
- **제조업 표준 용어**: "Overall Yield % = Good Units Produced / Total Units Started × 100" — "Good Units Produced"(정상품)는 판매/완제품으로 집계 가능한 수량을 가리키는 표준 용어이며, 불량품은 투입 자재·인건비를 소모했더라도 완제품 재고에는 포함되지 않는다.
- **MISys(MRP 소프트웨어) 사례**: 실제 수율이 계획과 다를 경우, 정상품 수량만 완제품 재고로 입고 처리되며, 부족분(스크랩으로 손실된 수량)은 별도로 재생산해야 원래 순소요량을 채울 수 있다고 명시.
- **결론**: "생산 시도 수(총 투입량)"는 계획/생산오더 수량일 뿐 재고 입고 수량이 아니며, 완제품 재고에는 항상 "정상품 수량(=생산시도수량×수율)"만 반영하는 것이 업계 표준. → **DECISIONS.md Q13 결정은 업계 관행과 부합하므로 유지 권장**. 다만 §5의 SPEC 원문 검증 기준(185/206 등)과의 잠재적 충돌은 여전히 남아있어 화면상 "생산시도수량"과 "정상품수량"을 함께 노출하는 방어적 설계는 유효.

## 4. (High) 실수(부동소수점) 연산 기반 수율·ceil 계산의 재현성 문제

**문제**: 수율은 실수(0.9 등)이고 부족분/수율에 ceil을 적용한다(§4). 부동소수점 나눗셈은 언어·플랫폼에 따라 미세한 오차가 발생할 수 있고(예: 100/0.9가 111.111...1 대신 111.99999999로 표현되는 경우), ceil 결과가 경계값에서 1 차이로 어긋날 수 있다.

**왜 실무에서 터지는가**: 정수/부동소수점 경계의 ceil 버그는 실제 생산관리 시스템에서 반복적으로 보고되는 클래식한 결함 유형이다. 특히 "딱 나누어떨어지는" 경계 케이스(예: 부족분=90, 수율=0.9 → 정확히 100)에서 부동소수점 오차로 인해 101이 나오는 사고가 흔하다.

**완화**: 수율을 정수 비율(예: 분자/분모, 또는 정수 퍼센트 1~100)로 내부 저장하거나, ceil 계산 시 정수 연산으로 치환하는 방식(정수 나눗셈의 올림 공식: (a + b - 1) / b 류)을 검토. 최소한 경계값(정확히 나누어떨어지는 케이스) 단위 테스트를 반드시 추가.

**업계 통상 채택 방식 (웹 리서치)**:
- **정수 올림 나눗셈 공식으로 치환**: 비율을 정수 스케일(예: 수율 0.9 → 90/100)로 변환한 뒤 `(부족분 × 분모 + 분자 - 1) / 분자` 형태의 순수 정수 연산으로 ceil을 계산 — 부동소수점을 아예 거치지 않아 오차가 원천적으로 없음.
- **C#/.NET `decimal` 타입 사용(가장 실용적인 채택안)**: `double`은 IEEE-754 이진 부동소수점이라 0.9를 정확히 표현할 수 없어 `90.0/0.9`가 `99.999999999999998` 또는 `100.00000000001`처럼 나올 수 있음. .NET의 `decimal`은 10진 고정소수점(28~29자리 정밀도)이며, Microsoft가 금융/수량 계산에 명시적으로 권장하는 타입. `decimal yield = 0.9m; Math.Ceiling(90m / yield)`는 정확히 100을 반환. Java의 `BigDecimal`도 동일한 이유로 `double`/`float` 대신 권장됨.
- **엡실론 보정(최후 수단)**: 부동소수점을 불가피하게 써야 한다면 `Math.Ceiling(a/b - epsilon)` 형태로 미세 오차를 흡수하는 방식도 있으나, 근본 해결책이 아닌 임시방편으로만 취급해야 함.
- **채택 권고**: 수율 필드를 `decimal` 타입으로 선언하고 ceil 계산도 `decimal` 연산으로 수행. `double`은 수량/금액 계산에 사용하지 않는다.

## 5. (High) PRODUCING 취소 불가(Q6) + REJECTED/RELEASE 종단 고정(Q8)이 만드는 "막다른 주문" 운영 리스크

**문제**: 한 번 PRODUCING에 들어간 주문은 취소할 방법이 없고, REJECTED/RELEASE는 영구 종단 상태다. 실제 운영에서는 "고객이 주문을 취소했는데 이미 생산 큐에 들어간 경우", "출고 후 오배송/불량 반품" 같은 상황이 반드시 발생하는데, 이 시스템은 이런 케이스에 대한 출구가 전혀 없다.

**왜 실무에서 터지는가**: SPEC 미명시 기능을 구현하지 않는다는 원칙(scope 최소화) 자체는 합리적이나, 이는 "기능이 없다"는 결정이지 "그런 상황이 발생하지 않는다"는 보장이 아니다. 실제 시연/채점 시나리오에서 "생산 중인 주문을 취소해보세요"라는 요청이 들어오면 시스템은 아무 대응도 못하고, 이것이 치명적 결함처럼 보일 수 있다.

**완화**: 최소한 "지원하지 않음"을 사용자에게 명확한 에러 메시지로 알리는 방어 코드가 필요하다(예: PRODUCING 상태 주문에 거절 명령을 시도하면 "생산 중인 주문은 취소할 수 없습니다"라는 명시적 안내). 조용히 무시되거나 알 수 없는 예외를 던지는 것은 UX·채점 모두에서 감점 요인이 된다.

**업계 통상 채택 방식 (웹 리서치)**:
- **TECO(Technically Complete) 상태(SAP 생산오더)**: 전체 삭제 없이 예약·소요량을 정지시키는 "논리적 종료" 상태. 취소가 아니라 통제된 상태 전이이며, 필요시 되돌리기(TECO 취소)도 가능하지만 그 전에 WIP/차이(variance)를 정산해야 함.
- **역방향 자재이동(reverse goods movement)을 통한 WIP 반제**: 이미 생산 확정(자재 소비)된 오더를 취소하려면 반제 이동유형(예: 262 — 출고 반제)으로 소비된 재고를 되돌린다. 부분취소도 확정수량의 일부만 반제하는 방식으로 지원됨.
- **상태 기반 취소 게이팅**: 실제 시스템은 오더 상태(생성/방출/확정/TECO/종료)에 따라 취소 가능 여부를 게이팅한다 — 자재·인건비가 투입되기 전(초기 상태)에는 완전 취소를 허용하고, 일단 투입되면 단순 "취소"가 아니라 별도의 반제(reversal) 워크플로로 안내한다.
- **취소 미지원 시 최소 UX**: 상태가 "시작 전"일 때만 즉시 취소 허용, "진행 중(WIP)"으로 넘어가면 취소 액션을 비활성화하거나 명시적 에러("이미 생산 중인 주문은 취소할 수 없습니다. 생산담당자에게 문의하세요")로 명확히 거부 — 조용히 실패하거나 상태를 오염시키지 않고 "닫힌 실패(fail closed)"로 처리하는 것이 표준.
- **채택 권고**: DECISIONS.md Q6 결정(PRODUCING 취소 미지원)은 업계 관행과 부합. 다만 반드시 명시적 거부 메시지를 구현에 포함시킬 것.

## 6. (Medium-High) 생산완료 판정 기준의 완전한 부재 (SPEC_ANALYSIS §2 PRODUCING→CONFIRMED 사후조건)

**문제**: "생산완료시" PRODUCING→CONFIRMED 전이가 일어난다고만 되어 있을 뿐, 무엇을 트리거로 완료를 판정하는지(총생산시간 경과? 사용자의 수동 명령? 콘솔 시간 흐름 시뮬레이션?) DECISIONS.md에서도 별도 Q로 다루지 않고 넘어갔다.

**왜 실무에서 터지는가**: 콘솔 애플리케이션은 실시간 타이머를 돌리기 어렵고(사용자 입력을 기다리는 blocking I/O가 대부분), "총생산시간이 지나면 자동 완료"를 구현하려면 백그라운드 스레드/타이머가 필요해 콘솔 앱 구조와 충돌한다. 반대로 "담당자가 수동으로 완료 명령을 입력"하는 방식으로 가면 §5-5.1 "생산현황표기"의 존재 의미(자동 갱신 현황판)와 어긋난다. 이 결정이 누락된 채로 구현이 진행되면 나중에 생산 메뉴 전체를 다시 설계해야 하는 재작업 리스크가 크다.

**완화**: Q11(순차 생산)과 마찬가지로 별도 Q17로 명시적 결정을 남길 것을 권장. 가장 현실적인 방향은 "생산담당자가 콘솔에서 수동으로 '생산완료 처리' 명령을 실행하면 큐의 선두 항목이 CONFIRMED로 전이"하는 수동 트리거 방식(콘솔 앱의 구조적 제약과 가장 잘 맞음).

**업계 통상 채택 방식 (웹 리서치)**: 블로킹 I/O 콘솔 앱(`Console.ReadLine()` 기반)에서는 실제 백그라운드 타이머/스레드보다 아래 방식이 훨씬 흔하고 안전함.
- **명시적 "틱(tick) 진행" 명령(시뮬레이션 시계)**: 운영자가 `생산진행` 같은 명령을 입력할 때마다 내부 시뮬레이션 시간 카운터가 증가하고, 누적 시간이 큐 선두 항목의 총생산시간에 도달하면 그 호출 안에서 즉시 PRODUCING→CONFIRMED 전이. 별도 스레드나 실제 시계가 필요 없고, 턴제 시뮬레이션 게임/공장 시뮬레이션 CLI에서 흔한 패턴.
- **수동 "완료 처리" 명령**: 운영자가 큐 선두 항목에 대해 `생산완료 처리` 명령을 직접 실행 — 경과시간 계산 자체를 하지 않고 운영자의 판단을 그대로 신뢰. 총생산시간은 화면에 "예상값"으로만 표시되고 실제 완료 트리거로는 쓰이지 않음(SPEC §4 검증절의 185 vs 206 불일치를 사이드스텝하는 효과도 있음).
- **다음 상호작용 시 지연 평가(lazy time-check)**: 별도 스레드 없이, 메인 입력 루프 안에서만(혹은 전용 `시간경과` 명령 실행 시에만) `DateTime.Now` 기준 경과시간을 계산해 완료 여부를 판정. `Timer`/`Thread.Sleep` 폴링을 쓰지 않으므로 `Console.ReadLine()` 블로킹과 충돌하지 않음.
- **채택 권고**: 이 프로젝트 구조(블로킹 콘솔 입력)에는 "수동 완료 처리 명령" 또는 "틱 진행 명령" 방식이 가장 적합. 백그라운드 `Timer`/스레드 방식은 도입 복잡도 대비 이득이 적어 지양.

## 7. (Medium) 참조 무결성 및 입력 검증의 사각지대 (존재하지 않는 시료ID로 주문, 중복 시료ID 등록)

**문제**: SPEC_ANALYSIS 1.2에서 이미 지적했듯, 시료예약(§5-2) 시 입력한 시료ID가 실제 등록된 시료인지 검증하는 규칙이 SPEC에 없고, DECISIONS.md도 이 부분(Q12는 수치 하한만 다룸)을 명시적으로 결정하지 않았다. 시료ID 자체의 유일성(중복 등록 방지)도 마찬가지로 결정되지 않았다.

**왜 실무에서 터지는가**: 이런 참조 무결성 누락은 "테스트 시나리오상 정상 흐름만 따라가면 안 드러나지만" 채점/QA 단계에서 "존재하지 않는 시료ID로 주문해보기", "같은 시료ID 두 번 등록해보기" 같은 엣지케이스 테스트에서 바로 드러나는 전형적인 취약 지점이다.

**완화**: Q12와 동일한 원칙(입력 시점 검증)을 확장 적용: 시료예약 시 시료ID 존재 여부 검증, 시료등록 시 시료ID 중복 검증을 명시적으로 결정·구현에 반영. 별도 Q로 문서화 권장.

**업계 통상 채택 방식 (웹 리서치)**:
- **참조 존재 검증(FK류 체크)**: 애플리케이션 계층에서 삽입/수정 전 참조 테이블을 조회해 존재 여부를 확인하는 것이 표준이며, 관계형 DB라면 여기에 DB 레벨 FK 제약을 최종 안전망으로 추가하는 "이중 방어"가 일반적. FK만으로는 앱 로직을 우회하는 스크립트·마이그레이션 실수를 막지 못하므로 앱 레벨 검증이 필수로 간주됨.
- **중복 방지(고유키 제약)**: 자연키(SKU/제품ID 등)에 `UNIQUE` 제약을 두는 것이 표준 — 동시 요청 상황에서는 앱 레벨 "조회 후 삽입" 로직만으로는 경쟁조건(race condition)이 발생하므로 DB 제약이 실제 원자성을 보장하는 유일한 방법. 중복 주문 제출 방지에는 "idempotency key"(클라이언트가 논리적 연산 단위로 고유키를 발급하고 서버가 이미 처리된 키인지 확인하는 방식)가 표준 패턴.
- **에러 처리 UX**: 명확한 거부 메시지(예: "제품ID X를 찾을 수 없습니다", "이미 존재하는 주문ID입니다")로 즉시 반려하는 것이 컨센서스 — 조용히 무시하거나 덮어쓰는 것은 데이터 문제를 숨기고 호출자의 기대를 깨뜨림.
- **채택 권고**: 시료예약 시 시료ID 존재 검증(앱 레벨), 시료등록 시 시료ID 중복 검증(딕셔너리/컬렉션의 키 존재 확인으로 충분 — 별도 DB 없이도 구현 가능)을 Q12와 동일한 "등록/입력 시점 검증" 원칙으로 추가할 것.

## 8. (Medium) 영속성 계층의 오류 복구 전략 부재 (파일 손상, 스키마 변경 시 하위호환)

**문제**: §7 "영속성"은 재실행 시 데이터 유지만 요구할 뿐, 저장 파일이 손상되었거나(비정상 종료 중 저장) 향후 엔티티 스키마가 바뀌었을 때(예: 주문ID 필드 추가) 기존 데이터를 어떻게 다루는지에 대한 규칙이 전혀 없다.

**왜 실무에서 터지는가**: 실제 현장에서 "영속성 버그"의 8할은 정상 저장/로드 경로가 아니라 비정상 종료(강제 종료, 디스크 꽉 참) 이후의 복구 경로에서 발생한다. 특히 이 시스템은 Q15에서 자동증가 정수 ID를 채택했는데, 파일이 손상되어 마지막 저장 시점을 알 수 없게 되면 다음 주문ID가 기존 주문과 충돌(중복 채번)할 위험이 있다.

**완화**: 저장은 "임시 파일에 쓰고 성공 시 교체(atomic rename)" 패턴을 적용해 저장 도중 프로세스가 죽어도 기존 파일이 손상되지 않도록 방어. 로드 시 파싱 실패에 대한 명확한 사용자 안내(예: "저장 파일이 손상되어 새로 시작합니다" 등) 정책을 미리 정해둘 것을 권장.

**업계 통상 채택 방식 (웹 리서치)**:
- **임시파일 작성 후 원자적 교체**: 대상 파일에 직접 쓰지 않고 같은 디렉터리(같은 볼륨)에 임시 파일(`data.json.tmp`)로 먼저 쓴 뒤 flush/fsync하고 교체. .NET에서는 `File.Replace(tempPath, targetPath, backupPath)`가 임시→대상 파일 원자적 교체와 "이전 파일을 backupPath로 보존"을 한 번에 처리 — Windows 환경에 특히 적합. 크로스플랫폼 대안으로 `File.Move(temp, target, overwrite: true)`(.NET Core 3.0+)도 있으나 `File.Replace`보다 원자성 보장이 약함.
- **백업 + 버전/체크섬**: `File.Replace`를 쓰지 않더라도 `data.json.bak`(직전 정상본)을 유지하고, JSON에 스키마 버전 필드와 체크섬(SHA-256/CRC32)을 포함시켜 파싱 성공 여부와 무관하게 손상(잘림 등)을 탐지.
- **로드 실패 시 복구 UX**: 기본 파일 로드 실패 → `.bak`으로 폴백(경고 표시: "복구된 이전 상태를 불러왔습니다") → 그마저 실패하면 손상 파일을 `.corrupt`로 보존한 채 새로 시작(진단용으로 삭제하지 않음), 명확한 경고와 함께 새 상태로 시작.
- **채택 권고**: `File.Replace` 기반 원자적 저장 + 로드 실패 시 폴백/경고 정책을 최소 구현으로 채택.

## 9. (Medium) 검색 기능(Q14)의 성능·모호성 트레이드오프

**문제**: Q14는 시료ID/이름/수율/평균생산시간을 모두 포함하는 통합검색으로 결정했으나, 숫자 속성(수율, 평균생산시간)에 대해 "정확 일치"와 "범위 일치" 중 무엇을 쓸지는 "구현 자유"로 남겨두었다. 이는 자유도가 아니라 미해결 모호성에 가깝다.

**왜 실무에서 터지는가**: 사용자가 "0.9"를 검색했을 때 수율이 정확히 0.9인 시료만 나오는지, 0.85~0.95 근처도 나오는지에 따라 체감 사용성이 완전히 달라진다. 채점자가 "수율로 검색해보세요"라는 시연을 요청했을 때 기대와 다른 결과가 나오면 "검색이 안 된다"는 인상을 줄 수 있다.

**완화**: 최소 기능으로 문자열(시료ID/이름) 부분일치 + 숫자 속성은 정확 일치로 우선 구현하고, 필요 시 범위 검색은 추가 기능으로 명시적으로 문서화(README)해 기대치를 관리.

**업계 통상 채택 방식 (웹 리서치)**:
- **정확 일치를 기본값으로(가장 저비용)**: 일반적인 CRUD/관리자용 검색 UI 다수는 숫자 필드도 문자열 필드와 동일하게 취급해 문자열 동등 비교로 매칭한다(예: "90" 입력 시 정확히 90인 값만 매칭, "0.9"나 85~95 범위는 매칭 안 됨). 이를 "숫자 필드는 정확 일치만 지원하며, 범위 검색은 별도 필터 패널을 사용하라"고 명시적으로 문서화하는 것이 일반적.
- **비교/범위 연산자 문법**: 더 정교한 검색 시스템은 통합 검색창에 연산자를 얹는다 — GitHub 검색 문법의 `field:>100`, `field:>=10`, `field:10..50`(`10..*`=이상, `*..10`=이하) 형태가 대표적이며, Jira JQL·Zendesk 등도 유사한 방식을 채택.
- **대괄호/구간 표기(검색엔진 백엔드)**: Lucene/OpenSearch/RediSearch는 `[min max]`(포함)/`(min max)`(제외) 구간 표기와 `+inf`/`-inf`를 사용하나, 이는 보통 사용자가 직접 타이핑하기보다 UI 필터 뒤에 숨겨진다.
- **채택 권고**: 이 규모의 콘솔 앱에는 "숫자 필드는 정확 일치만 지원"을 기본값으로 문서화(README)하고, 필요 시 별도 min/max 필터 명령을 추가 기능으로 남기는 것이 가장 현실적.

## 10. (Low-Medium) 감사 추적(audit trail)/타임스탬프 부재로 인한 사후 검증 불가

**문제**: SPEC_ANALYSIS 1.2에서 지적한 대로 주문의 "생성/전이 시각" 속성이 SPEC에 전혀 없다. FIFO 생산큐(Q11) 순서, 대기주문확인(§5-5.2) 화면의 "언제부터 대기 중인지" 등을 보여주려면 최소한의 시각 정보가 필요한데 이것이 누락되어 있다.

**왜 실무에서 터지는가**: 타임스탬프 없이 순서만 배열 인덱스(큐 위치)로 관리하면, 나중에 "이 주문이 언제 접수됐는지", "생산에 얼마나 걸렸는지"를 사후에 검증할 방법이 없다. 실무 MES에서는 이 정보가 없으면 SLA 위반, 지연 원인 분석 자체가 불가능해 운영 단계에서 반드시 요구되는 항목이다.

**완화**: 굳이 SPEC에 없더라도 내부 구현 상세로 각 주문에 생성 시각(및 상태 전이 시각)을 기록해두는 것을 권장. 화면 출력 의무는 없지만(§7 UI 자유), 추후 정렬·디버깅·로그 근거로 활용 가치가 크고 되돌리기 어려운 결정(Q15 자동증가 ID)과 달리 나중에 추가해도 비용이 낮으므로 지금 넣어두는 것이 안전하다.

**업계 통상 채택 방식 (웹 리서치)**:
- **감사 컬럼(Audit columns: CreatedAt/UpdatedAt, +CreatedBy/UpdatedBy)**: 주문·고객·재고 등 영속화되는 거의 모든 엔티티에 사실상 기본값으로 들어가는 필드. 스키마 설계 시점에 컬럼 하나·ORM 훅 하나 추가하는 비용이지만, 나중에 추가하려면 기존 레코드를 알 수 없는 값으로 백필하거나 이전 데이터의 공백을 그대로 받아들여야 함.
- **상태전이 이력 로그(append-only, "이벤트소싱 라이트")**: 현재 상태만 저장하는 대신 각 전이(RESERVED→CONFIRMED→RELEASE 등)를 불변 로그 행으로 기록. 완전한 이벤트소싱은 이벤트 재생으로 상태를 도출하지만, 경량 버전은 "현재 상태 테이블 + 이력 사이드 테이블" 조합으로 충분 — 결함/분쟁 발생 시 시퀀스 재구성(root-cause 분석)에 표준적으로 활용됨.
- **선제 도입의 비대칭적 이득**: 이 필드들은 초기에 추가해도 기존 비즈니스 로직에 영향이 없는 반면(additive), 일단 주문이 필드 없이 쌓인 뒤에는 "이 주문이 실제로 언제 접수/출고됐는지"를 영구히 복구할 수 없음 — 이 비대칭성(사전 비용 거의 0 vs 사후 영구 손실)이 표준 관행으로 자리잡은 근거.
- **채택 권고**: 최소한 CreatedAt(생성 시각)과 상태 전이 시각(간단한 append-only 리스트로도 충분)을 지금 구현에 포함시킬 것.
