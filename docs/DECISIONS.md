# DECISIONS

> 본 문서는 `docs/SPEC_ANALYSIS.md`의 OPEN_QUESTIONS(Q1~Q16)에 대해, 개발팀이 실제로 채택한 결정을 기록한다.
> SPEC에 명시되지 않은 사항은 웹 검색을 통한 통상적인 생산관리/재고관리(MRP 등) 업계 관행을 참고하여 보완하며, 각 결정에는 근거를 남긴다.

## Q1. "부족분"의 계산식

**결정**: 부족분 = max(0, 주문수 - 재고수량)

**근거**:
- SPEC 어디에도 "부족분"의 계산식이 명시되어 있지 않음(SPEC_ANALYSIS.md Q1).
- 일반적인 MRP(자재소요계획, Material Requirements Planning)의 표준 공식은 다음과 같음:
  - Net Requirement = Gross Requirement − (Scheduled Receipts + On Hand Inventory)
  - 결과가 음수가 되는 경우(재고가 충분한 경우) 0으로 처리(max(0, ...))하는 것이 통상적인 관행.
  - 참고: [Gross to Net MRP — The Formula Explained](https://craftybase.com/blog/gross-to-net-calculations-for-mrp/), [SAP MRP Step 6 - Calculation of Shortage Quantities](https://sites.google.com/site/sapswords/home/sap-mrp/sap-mrp-functionality/step-6---calculation-of-shortage-quantities-and-creation-of-procurement-proposals-in-sap-mrp)
- 이 시스템에는 "입고 예정량(Scheduled Receipts)" 개념이 SPEC에 없으므로, 단순화하여 `부족분 = max(0, 주문수 - 재고수량)`으로 채택.
- max(0, ...)로 하한을 고정함으로써, 재고가 주문수보다 많은 경우(SPEC_ANALYSIS.md §3.2 "음수" 항목)에도 실생산량 = ceil(0/수율) = 0으로 자연스럽게 처리되어 별도 분기 로직이 불필요해짐.

**영향**:
- §3.2 실생산량 = ceil(부족분/수율) 계산의 입력값이 확정됨.
- Q5(여유/부족/고갈 경계)와 연동: 부족분 > 0이면 "부족", 부족분 = 0이면서 재고수량 = 0이 아니면 "여유"로 판정 가능해짐(단, Q5는 별도 결정 필요).

## Q2. §4 검증 절 수식값 vs 화면표기값 불일치 (실생산량 185 vs 206, 총생산시간 148 vs 165)

**결정**: 수식값(실생산량=185, 총생산시간=148)을 정답으로 채택한다.

**근거**:
- SPEC이 명시한 계산 공식(실생산량 = ceil(부족분/수율), 총생산시간 = 평균생산시간 × 실생산량) 자체는 §4·§5·§6에서 반복적으로 명확히 서술되어 있음.
- 반면 화면표기 예시값(206, 165)은 §4 검증 절에서 SPEC 스스로 불일치를 인정한 지점이며, 어떤 보정 로직에서 유래했는지 근거가 SPEC에 전혀 없음.
- 수식은 재현 가능하고 검증 가능하지만, 표기값은 출처 불명의 숫자이므로 수식을 정답으로 채택하는 것이 시스템의 일관성과 테스트 가능성 측면에서 안전함.

**영향**:
- 실생산량/총생산시간 계산 로직 및 관련 단위 테스트는 SPEC 수식을 기준으로 작성한다.
- SPEC 문서상의 화면표기 예시(206, 165)는 SPEC 자체의 오기로 간주하고, 구현체의 화면 출력값도 수식 계산 결과(185, 148 등)를 그대로 노출한다.

## Q3. 수율=0일 때 실생산량 = ceil(부족분/수율) 0으로 나누기 문제

**결정**: 수율=0(또는 0 이하)은 시료등록 시점에 유효성 검증으로 거부한다. 즉 런타임(생산 계산 시점)에서 0으로 나누기 상황 자체가 발생하지 않도록 입력 단계에서 차단한다.

**근거**:
- 웹 리서치 결과 제조업 수율(yield)은 통상 0%~100% 범위로 정의되며(Good Units / Total Units Started), 0%는 "전량 불량"을 의미하는 극단값으로 이론적으로는 존재 가능하나, 정상적인 생산관리 시스템에서는 데이터 유효 범위를 등록/입력 시점에 검증하여 이상값이 하류 계산에 전파되지 않도록 하는 것이 일반적 관행임.
- SPEC_ANALYSIS.md Q4(수율 허용범위 미정)와 근본적으로 동일한 문제이므로, Q4의 결정(수율 ∈ (0, 1])과 일관되게 처리하는 것이 합리적. 즉 수율=0은 애초에 저장이 불가능하므로 §3.2의 0으로 나누기 상황 자체가 시스템 내에서 발생하지 않는다.
- 런타임(생산 실행) 시점에 예외 처리를 추가하는 대안은 "이미 잘못된 데이터가 저장된 이후 매번 방어 코드로 걸러야 하는" 구조이므로, 입력 단계 차단보다 코드 복잡도와 오류 발생 지점이 늘어남.

**영향**:
- 시료등록(§5-1) 입력 검증 로직에 "수율은 0 초과 1 이하(0 < 수율 ≤ 1)여야 한다"는 규칙 추가 (Q4 결정과 연동).
- 실생산량 계산 로직에서는 수율=0 케이스에 대한 별도 방어 코드(예외 처리)가 불필요해짐 — 해당 상태 자체가 도달 불가능(unreachable)함을 전제로 구현.

## Q4. 수율의 허용 범위

**결정**: 수율 ∈ (0, 1] (0 초과 1 이하)

**근거**:
- SPEC 예시(90/100=0.9)는 비율(실수, 0~1 스케일)로 수율을 표현하고 있으며, 100%(수율=1, 불량 없음)는 물리적으로 가능한 값이므로 상한 포함.
- 0%(수율=0, 전량 불량)는 Q3 결정에 따라 등록 자체를 거부하므로 하한은 배제(초과 조건).
- 수율이 1을 초과(정상 시료 수가 총 생산시료 수보다 많음)하는 것은 정의상 모순이므로 상한을 1로 고정.

**영향**:
- Q3 결정과 동일한 검증 로직으로 구현: 시료등록 시 `0 < 수율 ≤ 1`이 아니면 등록을 거부(입력 오류로 처리).

## Q5. "여유/부족/고갈" 재고상태를 가르는 수치 경계

**결정**:
- 고갈: 재고수량 = 0
- 부족: 0 < 재고수량 < 주문수 (해당 시료의 판단 대상 주문수 기준)
- 여유: 재고수량 ≥ 주문수

**근거**:
- 업계 관행(재주문점/최소재고 등)은 절대 임계값을 쓰는 경우가 많으나, 이는 별도의 "최소재고" 같은 속성이 SPEC에 정의되어 있을 때 성립하는 방식이며, 본 SPEC에는 그런 속성이 없음.
- SPEC_ANALYSIS.md §3.4에서 이미 지적했듯, 이 판정이 "주문승인" 시의 재고충분/부족 판단(§3, §5-3)과 동일 기준을 공유하는지가 SPEC에 없는 문제였음 → 동일 기준을 공유하는 것으로 결정하여 Q1(부족분 = max(0, 주문수-재고수량))과 정합성을 맞춤: 부족분 > 0이면 "부족", 재고수량=0이면 "고갈"(부족분>0인 경우를 포괄하되 더 구체적인 케이스), 그 외(부족분=0이고 재고수량>0)면 "여유".
- 재고수량 자체에 대한 절대 기준(예: 5개 미만이면 부족)으로 정의할 경우, "부족" 상태가 특정 주문과 무관하게 존재해야 하는데 SPEC은 이런 절대 임계값을 어디에도 제시하지 않음. 반면 "주문수 대비"로 정의하면 SPEC이 유일하게 제공하는 수치(주문수)를 그대로 재사용할 수 있어 근거가 더 명확함.

**영향**:
- 모니터링(§5-4.2 재고량확인) 화면은 시료별로 해당 시료에 걸려 있는 활성 주문(RESERVED 등 판단 대상)의 주문수를 기준으로 여유/부족/고갈을 표기한다. 특정 주문과 무관한 일반 조회 시(예: 시료조회 §5-1.2)에는 이 판정을 적용하지 않고 재고수량만 노출한다.
- Q9(승인 시 재고 차감 시점)와 함께, "재고충분/부족" 판단 로직이 시스템 전체에서 일관된 단일 함수로 구현되어야 함.

## Q6. PRODUCING 상태 주문의 취소(거절) 가능 여부

**결정**: 미지원. PRODUCING → REJECTED 전이는 구현하지 않는다.

**근거**:
- 웹 리서치 결과, 실제 제조 시스템(ERP/MES)에서도 생산 중(In progress) 작업 지시를 취소하는 것은 "이미 투입된 자재·예약된 설비·큐 스케줄을 되돌리는" 복잡한 절차이며(예: SAP의 생산오더 취소 시 자재 소비 반제(reversal) 로직 필요), 단순한 상태값 변경 이상의 부수 처리가 필요함.
- 본 SPEC은 생산큐 항목 제거, 이미 소모된 자원(생산라인 점유 등)의 반환 규칙을 전혀 정의하지 않았으므로, 이런 되돌림 로직을 임의로 설계해 추가하면 SPEC에 없는 정합성 규칙을 시스템이 자체적으로 발명하게 되는 위험이 큼.
- 원칙적으로 SPEC에 명시되지 않은 기능은 구현하지 않는다(scope 최소화) — 미지원 쪽이 더 안전하고 이후 SPEC이 명확해지면 추가하기 쉬움.

**영향**:
- 상태 전이표(§2)에 PRODUCING → REJECTED 전이는 추가하지 않는다.
- 생산 중인 주문은 §3에서 정의된 대로 PRODUCING → CONFIRMED 전이만 가능하다.

## Q7. CONFIRMED(출고 대기중) 상태 주문의 출고 전 취소 가능 여부

**결정**: 허용. CONFIRMED → REJECTED 전이를 추가하며, 취소 시 승인 단계에서 차감된 재고를 복구한다.

**근거**:
- CONFIRMED는 생산이 이미 끝났거나(원래 재고충분) 필요 없던 상태로, PRODUCING과 달리 "이미 투입된 자재나 진행 중인 물리적 생산 공정"이 없어 취소에 따른 부수 효과가 재고 수치 복구 하나로 한정됨 → Q6(PRODUCING 취소, 미지원)과 달리 되돌림 로직이 단순함.
- 실무 주문관리 시스템에서도 "출고/배송 전 취소"는 일반적으로 지원되는 기능이며, 이 시점의 취소는 재고 예약 해제 정도로 처리 가능함(웹 리서치: shop order 취소 시 "reservations ... are also canceled, making the ... material reserved ... available for other shop orders").
- 단, 이 결정은 Q9(재고 차감 시점)의 결정에 의존한다: 재고가 승인 시점에 이미 차감되어 있어야 "복구"할 대상이 존재함.

**영향**:
- 상태 전이표(§2)에 CONFIRMED → REJECTED 전이 추가.
- 취소(거절) 처리 로직은 Q9에서 결정된 차감 시점 기준으로, 해당 주문의 주문수만큼 재고수량을 되돌린다.
- REJECTED는 여전히 최종 상태(더 이상 전이 없음)로 유지한다(Q8 참고).

## Q8. RELEASE가 최종 상태인지 (반품/재입고 지원 여부)

**결정**: RELEASE는 최종 상태로 확정한다. 반품/재입고 전이는 지원하지 않는다.

**근거**:
- Q6과 동일한 원칙: SPEC에 명시되지 않은 기능(반품, 재입고 시 재고/상태 처리 규칙)은 구현하지 않는다.
- RELEASE는 물리적으로 시료가 이미 고객에게 전달된 상태를 의미하므로, 반품 처리는 재고 복구뿐 아니라 품질 검사, 부분 반품 등 SPEC이 전혀 다루지 않는 추가 규칙이 필요해 scope를 크게 벗어남.

**영향**:
- 상태 전이표(§2)에서 RELEASE는 out-degree 0인 종단 상태로 확정한다.
- REJECTED, RELEASE 두 상태 모두 시스템상 최종(terminal) 상태로 취급한다.

## Q9. 재고 차감 시점 (주문승인 시 즉시 vs 출고 시 재확인)

**결정**: 주문승인(RESERVED→CONFIRMED 또는 RESERVED→PRODUCING) 시점에 재고수량에서 주문수를 즉시 차감한다.

**근거**:
- 승인 시점에 차감하지 않으면, 여러 주문이 동시에 "재고충분"으로 오판되어 실제 재고보다 많은 수량이 CONFIRMED 처리되는 오버부킹이 발생할 수 있음(Q10과 직결). 즉시 차감은 이 문제를 근본적으로 방지하는 가장 단순한 방법.
- Q7에서 결정한 "CONFIRMED→REJECTED 취소 시 재고 복구" 로직이 성립하려면, 애초에 승인 시점에 재고가 차감되어 있어야 복구할 대상이 존재함 → Q7 결정과 논리적으로 연결됨.
- SPEC §5-6의 "재고가 충분해진 CONFIRMED 주문을 출고 처리"라는 표현은, 승인 시점에 이미 재고가 차감(예약)된 상태에서 생산 완료로 재고가 채워져 "충분해지는" 것으로 해석 가능하며 모순되지 않음.

**영향**:
- 주문승인(§5-3) 로직: CONFIRMED/PRODUCING으로 전이하는 순간 재고수량 -= 주문수를 실행한다.
- 생산완료(PRODUCING→CONFIRMED) 시점에는 생산된 실생산량만큼 재고수량을 증가시킨다(§5-5/§6).
- 출고(CONFIRMED→RELEASE) 시점의 "재고가 충분해진" 조건은, 이미 예약 차감된 값이 생산 완료로 다시 채워졌는지를 확인하는 재검증으로 해석한다.

## Q10. 여러 주문이 동일 시료에 대해 동시에 승인 요청될 때의 순서/동시성 규칙

**결정**: 단일 프로세스 내 순차 처리로 해결한다. 멀티 프로세스/멀티유저 동시 접근(예: 프로그램을 두 번 동시에 실행해 같은 저장소에 동시에 쓰기)은 SPEC 범위 밖(known limitation)으로 명시하고 별도 방어 로직(분산 락 등)은 구현하지 않는다.

**근거**:
- SPEC §7 "영속성"은 "application을 다시 실행해도 데이터를 유지할 수 있는 성질"로 정의되어 있으며, 이는 재시작 간 데이터 보존을 의미할 뿐 멀티 프로세스/멀티유저 동시 접근을 요구하지 않음.
- 시스템은 콘솔 기반 단일 사용자 애플리케이션으로 설계되어 있어, 한 프로세스 안에서는 사용자가 한 번에 하나의 명령만 입력하므로 자연히 요청이 순차적으로 처리됨. 이 안에서는 Q9(승인 시 즉시 재고 차감)만으로 오버부킹이 방지됨.
- 동일 데이터 저장소에 대한 멀티 프로세스 동시 쓰기 방어(파일 락, DB 트랜잭션 격리 수준 등)는 SPEC이 전혀 요구하지 않는 별도의 인프라 수준 과제이며, 학생 프로젝트 성격의 콘솔 앱 범위를 벗어나는 과잉설계(overengineering)로 판단됨.

**영향**:
- 주문승인 로직은 별도의 락(lock) 없이, 요청이 들어온 순서대로 처리하는 것으로 구현한다(단일 프로세스 내 순차성 전제).
- README/설계 문서에 "본 시스템은 단일 프로세스 실행을 전제로 하며, 동일 데이터 저장소에 대한 멀티 프로세스 동시 접근은 지원 범위에서 제외한다"는 제약사항을 명시한다.

## Q11. 생산큐(FIFO)의 순차/병렬 생산 여부

**결정**: 순차 생산(단일 생산라인). 생산큐의 항목은 하나의 생산라인에서 한 번에 하나씩만 처리된다.

**근거**:
- SPEC에는 "생산라인"(단수)이라는 표현만 등장하고 라인 개수·병렬 capacity 개념이 전혀 없으므로, 병렬 생산을 지원하려면 SPEC에 없는 개념(라인 수, 자원 배정 규칙)을 임의로 발명해야 함.
- 단일 라인 순차 처리로 정하면 완료 시점 예측이 명확해짐: 특정 큐 항목의 예상 완료 시점 = 자신보다 앞선 모든 항목의 총생산시간 누적 + 자신의 총생산시간. 이는 "대기주문확인"(§5-5) 화면에서 대기 순서를 보여주는 목적과 자연스럽게 부합.
- 병렬 생산을 허용하면 "대기 중"이라는 개념 자체가 모호해짐(여러 항목이 동시에 진행 중이면 무엇이 "대기"인지 불명확).

**영향**:
- 생산큐는 FIFO 순서를 유지하는 단일 큐로 구현하고, 한 번에 하나의 항목만 PRODUCING 실행 상태로 처리한다.
- 대기주문확인(§5-5.2) 화면은 큐에 남은 항목과 그 순서를 그대로 노출하면 되며, 병렬 처리로 인한 순서 역전 문제를 고려할 필요가 없다.

## Q12. 수치 속성(주문수/재고수량/평균생산시간)의 하한/자료형 검증 규칙

**결정**: 등록/입력 시점에 검증한다.
- 주문수 > 0 (양의 정수)
- 재고수량 ≥ 0 (음이 아닌 정수)
- 평균생산시간 > 0 (양수)

**근거**:
- 웹 리서치 결과, 일반적인 주문관리 시스템의 검증 관행은 "자료형(숫자 여부) + 범위(주문수량에 허용 가능한 값) + 정책 정합성"을 주문/등록 시점에 자동 검증하는 것임(예: minimum order quantity 검증, order validation procedure에서 수량 필드 사전 검증).
- Q3/Q4에서 이미 확립한 원칙(수율=0/범위밖 값은 등록 시점에 거부)과 동일한 논리를 다른 수치 속성에도 일관되게 적용: 물리적으로 의미 없는 값(음수 주문량, 0 이하 생산시간)이 저장되면 이후 모든 계산식이 무의미한 값을 산출하는 문제(SPEC_ANALYSIS.md Q12에서 지적)를 입력 단계 검증으로 원천 차단.
- 재고수량은 0을 포함해야 함(§3.4 "고갈"이 재고수량=0으로 정의되므로 0은 유효한 정상 상태).

**영향**:
- 시료등록(§5-1.1) 입력 검증: 평균생산시간 > 0.
- 시료예약(§5-2) 입력 검증: 주문수 > 0(정수).
- 재고수량은 시스템 내부 로직(승인 시 차감, 생산완료 시 증가)에 의해서만 변경되며, 항상 0 이상을 유지하도록 하는 불변식을 코드 레벨에서 보장한다(Q1의 max(0, ...) 설계와 일관).

## Q13. 실생산량 중 재고에 반영되는 수량 (실생산량 전체 vs 실생산량×수율)

**결정**: 실생산량 × 수율(내림 또는 정수화 규칙은 별도 결정 필요)을 재고에 반영한다. 즉 "실생산량"은 "생산 시도 수(총 투입량)"를 의미하며, 이 중 수율만큼만 정상품으로 재고에 더해진다.

**근거**:
- 웹 리서치 결과, 제조업 표준 용어에서 "Total units started/produced"(생산 시도/투입 수량)와 "Good units produced"(정상품/양품 수량)는 명확히 구분되며, 완제품 재고(finished goods inventory)에 반영되는 것은 항상 후자(Good Units = Total Units × Yield)임.
- "실생산량 = ceil(부족분/수율)"이라는 SPEC의 수식 자체가 "수율 손실을 감안해 부족분만큼의 정상품을 확보하려면 얼마나 많은 양을 생산 시도해야 하는가"를 계산하는 역산 공식이므로, 실생산량은 "시도 횟수/투입량"의 의미로 봐야 정합적임. 만약 실생산량 전체를 그대로 재고에 반영하면 ceil과 수율 나눗셈 자체가 무의미해짐(수율이 얼마든 상관없이 항상 부족분 이상이 채워지므로 애초에 수율로 나눌 이유가 없어짐).
- ceil 연산으로 인해 실생산량×수율은 부족분보다 같거나 약간 큰 값이 되어(예: 부족분=100, 수율=0.9 → 실생산량=ceil(100/0.9)=112 → 정상품=112×0.9=100.8), 부족분을 충족하는 최소 시도 수를 계산하는 SPEC의 의도와 부합함.

**영향**:
- 생산완료(PRODUCING→CONFIRMED) 처리 시, 재고수량 += floor(실생산량 × 수율) (정수화 방식은 반올림 규칙이 SPEC에 없으므로 재고가 실제보다 부풀려지지 않도록 내림(floor)을 기본으로 채택 — 추가 논의 필요 시 별도 Q로 분리 가능).
- "실생산량"이라는 용어는 화면 표기 시 "생산 시도 수량"의 의미로 사용하고, 필요 시 별도로 "정상품 수량(=실생산량×수율)"을 함께 노출하는 것을 고려한다.

## Q14. 시료검색(§5-1.3) "속성으로 검색"의 대상 범위

**결정**: 시료ID, 이름, 수율, 평균생산시간을 모두 포함하는 통합검색으로 구현한다.

**근거**:
- SPEC이 "속성 기반 검색"이라고만 서술하고 특정 속성으로 한정하지 않았으므로, 시료 엔티티(§1.1)에 정의된 모든 속성(재고수량 제외 — 이는 파생/조회 전용 값이라 검색 키라기보다 결과 표시 항목에 가까움)을 검색 대상으로 포함하는 것이 SPEC의 "속성으로 검색"이라는 포괄적 문구에 가장 부합함.
- 이름만으로 한정하면 SPEC이 굳이 "이름검색"이라 하지 않고 "속성검색"이라고 표현한 이유를 설명하지 못함.

**영향**:
- 시료검색 기능은 입력된 검색어를 시료ID/이름(문자열 부분일치), 수율/평균생산시간(숫자 값 또는 범위 일치)에 대해 교차 검색하여 결과를 반환한다.
- 구체적인 숫자 속성(수율, 평균생산시간) 검색 시 정확 일치/범위 검색 여부 등 세부 UX는 구현 자유(§7 "UI 자유" 원칙)로 남긴다.

## Q15. 주문ID 채번 방식

**결정**: 자동증가 정수(1, 2, 3, ...)를 사용한다.

**근거**:
- 본 시스템은 단일 프로세스 콘솔 애플리케이션(Q10 결정 참고)이므로 분산 환경에서의 ID 충돌을 걱정할 필요가 없어 UUID의 이점(전역 고유성)이 불필요함.
- 자동증가 정수는 구현이 가장 단순하고, 콘솔에서 사용자가 "특정 주문"을 지칭할 때(§3, §5-3, §6) 입력하기도 쉬우며, 생성 순서를 그대로 보여주어 FIFO 생산큐 순서(Q11)와도 개념적으로 자연스럽게 연결됨.

**영향**:
- 주문 엔티티(§1.2)에 정수형 주문ID 속성을 추가하고, 시료예약(§5-2) 시 시스템이 자동으로 다음 순번을 부여한다.
- 콘솔 명령에서 "특정 주문"을 지정할 때는 이 정수 ID를 인자로 사용한다.

**보강 (2026-07-15, ADR-E4/E5 논의 중 확인)**: SPEC의 화면 예시(§5-2 등)에 등장할 수 있는 `ORD-YYYYMMDD-NNNN` 형태의 표기는 규범적 채번 규칙이 아니다. SPEC은 화면 구성을 자유(§7 "UI 자유")로 명시했으므로, 예시 UI에 등장하는 문자열은 Q2가 이미 확립한 원칙("예시 UI의 숫자/표기는 규범이 아니며, SPEC의 수식·규칙 조문이 정답이다")과 동일하게 취급한다. 따라서 자동증가 정수 채번(위 결정)은 이 예시 표기와 충돌하지 않으며 그대로 유지한다. [ADR-Q2]와 상호 참조. 관련: [[ADR-E4]], [[ADR-E5]].

## Q16. 곱셈 기호 "×" 표기 적용 여부

**결정**: 화면 표기에 "×" 기호를 적용한다.

**근거**:
- SPEC §4 각주에서 명시적으로 "×"로 표기하라는 지시가 존재하며, 정식 UI 섹션(§5, §6)에 반복되지 않은 것은 SPEC 문서의 구조적 중복 누락(SPEC_ANALYSIS.md에서 지적된 §6의 중복 서술 패턴)일 뿐 지시 자체가 철회된 것은 아님.
- 명시적으로 존재하는 지시를 따르는 것이 안전하며, 화면 가독성 측면에서도 "×"가 "*"보다 일반 사용자에게 더 친숙한 표기임.

**영향**:
- 총생산시간 계산 결과를 화면에 표기할 때 "평균생산시간 × 실생산량 = 총생산시간" 형태로 출력한다.

## R1. 단일 인스턴스 가드 (RISKS §1 → PRD FR-24)

**결정**: 프로그램 실행 시 동일 데이터 저장소를 대상으로 이미 실행 중인 인스턴스가 있으면 경고 메시지를 표시하고 즉시 종료한다.

**선택안 (웹 리서치)**:
- **선택안 A — OS 네임드 뮤텍스(Windows)**: 전역 이름(GUID 포함)의 Named Mutex를 생성하고 `createdNew == false`이면 즉시 종료. 프로세스 비정상 종료 시 OS가 자동 해제하여 stale lock 정리 로직이 불필요.
- **선택안 B — 배타적 파일 잠금**: 저장 파일을 앱 생애주기 동안 배타 오픈(`FileShare.None` 상당)으로 유지. 두 번째 프로세스가 열려고 하면 즉시 예외 발생. 이 역시 프로세스 종료 시 자동 해제.
- **선택안 C — PID 기반 사이드카 lock 파일**: lock 파일에 PID 기록 후 해당 프로세스 생존 여부 확인. 크래시 시 orphan lock이 남는 문제로 구현이 번거로움.
- **채택**: 선택안 A(Named Mutex 상당) 또는 B(배타적 파일 잠금) — 이 프로젝트(C++20 콘솔, MSVC/Win32)에서는 Windows `CreateMutexW`(전역 이름) 또는 저장 파일 배타 오픈 중 구현 난이도가 낮은 쪽을 채택. C는 배제.

**근거**:
- Q10(멀티 프로세스 동시 접근은 SPEC 범위 밖, 문서상 제약)은 "결정"일 뿐 코드로 강제되지 않아, 실제로는 프로그램을 두 번 실행하면 Q10이 우려한 오버부킹/데이터 덮어쓰기가 그대로 재현된다.
- Named Mutex/배타적 파일 잠금은 모두 크래시 시 OS가 자동 해제하므로 별도의 "정상 종료 보장" 로직이 필요 없어, 이 프로젝트 규모에 가장 적합.

**영향**: PRD FR-24. Q10의 "문서상 제약"을 "코드로 강제되는 제약"으로 격상.

**테스트를 위한 보강**:
- 프로그램을 정상 실행 중인 상태에서 동일 실행 파일을 한 번 더 기동 → 두 번째 실행이 경고 메시지 출력 후 즉시 종료하는지 검증.
- 첫 번째 인스턴스를 정상 종료한 뒤 재실행 → 정상적으로 기동되는지(락이 잔류하지 않는지) 검증.
- (가능하다면) 강제 종료(비정상 종료) 후 재실행 → 잔류 락으로 인해 정상 기동이 막히지 않는지 검증.

## R2. 상태전이·재고증감의 원자적 처리 (RISKS §2 → PRD FR-25)

**결정**: 주문 상태 전이와 재고 증감(T1, T2, T4, T6)을 하나의 논리적 단위(단일 메서드 호출)로 묶고, 저장 실패 시 인메모리 변경도 롤백한다.

**선택안 (웹 리서치)**:
- **선택안 A — 단일 원자적 메서드 + 임시파일 교체 저장**: 검증 → 인메모리 상태전이+재고증감 → 전체 상태를 임시파일에 쓴 뒤 rename(동일 볼륨 내 원자적 교체)으로 flush. 순서를 "검증 먼저, flush 마지막"으로 고정.
- **선택안 B — Command 패턴 + 인메모리 롤백**: 각 전이를 before/after를 가진 커맨드 객체로 캡슐화, 실패 시 역순 되돌림.
- **선택안 C — Write-Ahead Log(WAL)**: 의도 기록 → 적용 → 커밋 표시, 재시작 시 미완료 항목 재생/폐기.
- **채택**: 선택안 A. B(과설계 — 필드 2개만 얽힌 이 프로젝트엔 불필요)와 C(콘솔 앱 규모 대비 overkill)는 배제.

**근거**:
- Q9(승인 시 즉시 차감)와 Q7(취소 시 복구)가 모두 "재고수량"이라는 동일 공유 상태를 수정하는데, 두 단계 쓰기 중 하나만 성공하는 부분 실패가 실제 MES/ERP 사고의 다수를 차지한다.
- 이 프로젝트 규모(필드 2개: 상태, 재고수량)에서는 단일 원자적 메서드가 정합성과 구현 비용의 균형점.

**영향**: PRD FR-25. T1/T2/T4/T6 구현 시 각 전이 로직이 단일 진입점(예: `ApproveOrder`, `CompleteProduction` 등)에서 검증→변경→저장을 한 번에 처리하도록 설계.

**테스트를 위한 보강**:
- 상태전이 도중(재고 차감 후 상태 저장 전) 예외를 강제로 발생시키는 테스트 → 재고수량이 전이 이전 값으로 유지되는지 검증(단위 테스트 필수 항목).
- T1(RESERVED→CONFIRMED), T2(RESERVED→PRODUCING), T4(PRODUCING→CONFIRMED), T6(CONFIRMED→REJECTED) 각각에 대해 "저장 실패 시 재고수량 불변" 케이스를 개별 작성.

## R3. Q13(수율 이중 적용) 해석의 업계 정합성 재확인 (RISKS §3)

**결정**: 기존 Q13 결정(실생산량×수율만 재고 반영, 정상품 기준)을 그대로 유지한다. 다만 화면에는 "생산시도수량(실생산량)"과 "정상품수량(재고반영분)"을 함께 노출하는 방어적 표시를 권장한다.

**선택안 (웹 리서치)**:
- 재확인 결과(선택안이 아닌 검증): SAP PP/MRP의 scrap factor 로직은 순소요량을 스크랩률만큼 부풀린 "계획 오더 시작 수량"으로 산출하되, 가용재고/순소요량 계산은 항상 정상품(Good Units) 기준으로만 이루어짐. "Overall Yield % = Good Units Produced / Total Units Started"가 업계 표준 용어. MISys 등 MRP 소프트웨어도 정상품 수량만 완제품 재고로 입고 처리.
- **결론**: Q13 결정은 업계 표준과 일치 — 별도 재결정 불필요, 유지.

**근거**:
- SPEC §4 검증 절의 예시값(206/165)이 Q2에서 이미 "SPEC 오기"로 기각되었으나, 화면표기값이 실생산량을 그대로 재고에 반영하는 뉘앙스로 읽힐 여지가 있어 채점/인수 단계에서 다른 멘탈모델과 충돌할 위험이 남아있음.
- 웹 리서치로 실무 표준과 부합함을 확인했으므로 로직 자체는 변경하지 않되, 화면 표기를 통해 어느 멘탈모델로 검증하든 답을 확인할 수 있게 방어.

**영향**: 별도 신규 FR 없음(FR-19의 화면 표시 방식에 대한 보강 권고로, PRD FR-19 구현 시 참고). Q13 결정 자체는 변경 없음.

**테스트를 위한 보강**:
- 생산완료 처리 후 화면에 "생산시도수량"과 "정상품수량(재고반영분)"이 둘 다 표시되는지 확인하는 UI 레벨 확인 항목(자동화 단위 테스트보다는 `/verify` 수동 확인 대상).
- 계산 로직 자체는 기존 Q13/FR-17~19 테스트 케이스로 충분(신규 계산 테스트 불필요).

## R4. 수율/ceil 계산의 부동소수점 재현성 (RISKS §4 → PRD FR-26)

**결정**: 수율 및 실생산량/재고반영량 계산에 부동소수점(`double`/`float`) 대신 정수 또는 고정소수점 기반 연산을 사용한다.

**선택안 (웹 리서치)**:
- **선택안 A — 정수 올림 나눗셈 공식 치환**: 수율을 정수 스케일(예: 0.9 → 90/100)로 변환 후 `(부족분 × 분모 + 분자 - 1) / 분자` 형태의 순수 정수 연산으로 ceil 계산. 부동소수점을 아예 거치지 않음.
- **선택안 B — `decimal`/`BigDecimal`류 고정소수점 타입**: 10진 고정소수점 타입으로 수율을 표현하고 해당 타입 연산으로 ceil 계산. C++20에는 표준 `decimal` 타입이 없으므로 이 프로젝트에는 직접 적용 불가.
- **선택안 C — 엡실론 보정**: `ceil(a/b - epsilon)` 형태로 오차 흡수. 근본 해결책이 아닌 임시방편.
- **채택**: 선택안 A(정수 스케일 연산). C++20 환경에서는 `decimal` 표준 타입이 없으므로, 수율을 분자/분모 정수 쌍(또는 정수 퍼센트 1~100)으로 내부 저장하고 정수 올림 나눗셈 공식으로 ceil을 계산한다. C(엡실론 보정)는 임시방편이므로 배제.

**근거**:
- "딱 나누어떨어지는" 경계 케이스(예: 부족분 90, 수율 0.9 → 정확히 100)에서 IEEE-754 이진 부동소수점 오차로 101이 나오는 사고가 실제 생산관리 시스템에서 반복 보고되는 클래식 결함.
- C++20에는 .NET `decimal` 같은 표준 10진 고정소수점 타입이 없어 선택안 B는 이 스택에 그대로 적용 불가 — 정수 연산 치환이 언어 제약상 가장 현실적인 대안.

**영향**: PRD FR-26. Sample의 수율 저장 방식(내부 표현)이 단순 `double` 필드가 아니라 정수 분자/분모(또는 정수 퍼센트) 기반으로 재설계될 수 있음 — Model 설계 시 반영 필요.

**테스트를 위한 보강**:
- 경계값 단위 테스트 필수 추가: 부족분/수율이 정확히 나누어떨어지는 케이스(예: 부족분=90, 수율=0.9 → 실생산량=100)에서 정수 연산 결과가 정확히 100인지 검증.
- 나누어떨어지지 않는 일반 케이스(예: 부족분=170, 수율=0.92 → 실생산량=185, SPEC §4 근거값)도 동일 경로로 재검증.
- `double` 기반 순수 부동소수점 연산과 비교하는 회귀 테스트를 추가해, 향후 리팩터링 시 다시 `double`로 되돌아가는 회귀를 방지.

## R5. 지원하지 않는 전이 시도 시 명시적 거부 (RISKS §5 → PRD FR-27)

**결정**: PRODUCING 상태 주문에 대한 거절 시도, REJECTED/RELEASE 상태 주문에 대한 모든 전이 시도는 조용히 무시하지 않고 명시적 거부 메시지를 표시한다.

**선택안 (웹 리서치)**:
- **선택안 A — 상태 기반 취소 게이팅 + 명시적 거부 메시지("fail closed")**: 오더 상태에 따라 취소 가능 여부를 게이팅하고, 진행 중(WIP) 이후에는 취소 액션을 비활성화하거나 명시적 에러로 거부.
- **선택안 B — TECO(Technically Complete) 방식 되돌리기 지원**: 취소가 아닌 통제된 상태 전이로, 필요시 되돌리기도 지원하되 WIP/차이 정산 필요.
- **선택안 C — 역방향 자재이동(reverse goods movement)을 통한 부분 반제**: 이미 소비된 자재의 일부만 되돌리는 정교한 반제 워크플로.
- **채택**: 선택안 A. B/C는 Q6/Q8에서 이미 "SPEC에 없는 되돌림 로직은 구현하지 않는다"고 결정한 scope 밖이므로 배제 — 이번 R5는 되돌림 기능 추가가 아니라 "미지원을 조용히 무시하지 않고 명확히 알리는" UX 보강에 한정.

**근거**:
- Q6(PRODUCING 취소 미지원), Q8(REJECTED/RELEASE 종단 고정) 자체는 "기능이 없다"는 결정이지 "그런 시도가 발생하지 않는다"는 보장이 아니다. 실제 시연/채점 시나리오에서 이런 시도가 들어오면 시스템이 아무 반응 없이 무시하거나 알 수 없는 예외를 던지는 것은 UX·채점 모두에서 감점 요인.
- "fail closed"(명확한 거부)는 조용한 실패보다 항상 안전하며, Q6/Q8의 scope 자체를 넓히지 않으므로 기존 결정과 충돌하지 않는다.

**영향**: PRD FR-27(FR-13 보강). Controller 계층에서 PRODUCING/REJECTED/RELEASE 상태에 대한 미지원 전이 요청을 가로채 View를 통해 거부 메시지를 출력하도록 구현(§2 아키텍처 규칙 — Controller는 View를 통해서만 출력).

**테스트를 위한 보강**:
- PRODUCING 상태 주문에 거절(주문거절) 명령을 시도 → 상태가 변경되지 않고 명시적 거부 메시지가 반환되는지 검증.
- REJECTED 상태 주문에 임의의 전이(승인/거절/취소/출고)를 시도 → 모두 거부되고 상태 불변인지 검증.
- RELEASE 상태 주문에 임의의 전이를 시도 → 동일하게 거부되고 상태 불변인지 검증.

## R6. 생산완료 처리 트리거 확정 — Q17 (RISKS §6 → PRD FR-28)

**결정**: PRODUCING → CONFIRMED 전이(생산완료)는 생산담당자의 수동 명령("생산완료 처리")으로 트리거된다. 백그라운드 타이머/스레드 방식은 채택하지 않는다.

**선택안 (웹 리서치)**:
- **선택안 A — 수동 "완료 처리" 명령**: 운영자가 큐 선두 항목에 대해 "생산완료 처리" 명령을 직접 실행. 경과시간 계산 없이 운영자 판단을 신뢰. 총생산시간은 화면상 "예상값"으로만 표시.
- **선택안 B — 명시적 "틱(tick) 진행" 명령(시뮬레이션 시계)**: 운영자가 진행 명령을 입력할 때마다 내부 시뮬레이션 시간이 누적되고, 총생산시간 도달 시 자동 전이.
- **선택안 C — 다음 상호작용 시 지연 평가(lazy time-check)**: 전용 명령 실행 시에만 실제 경과시간을 계산해 완료 여부 판정.
- **선택안 D — 백그라운드 Timer/Thread**: 실시간 타이머로 자동 완료. 블로킹 콘솔 입력(`std::cin` 등) 구조와 충돌하고 도입 복잡도 대비 이득이 적어 배제.
- **채택**: 선택안 A(수동 완료 처리 명령). 이 프로젝트의 콘솔 입출력 구조(블로킹 입력) 및 §5-5 "생산현황표기"·§5-5.2 "대기주문확인" 화면과 가장 단순하게 부합.

**근거**:
- 콘솔 앱은 블로킹 I/O가 대부분이라 실시간 타이머(D)는 구조적으로 맞지 않음. B/C는 "경과시간 자동 판정"을 요구하는데, SPEC §4의 185 vs 206 수식-표기 불일치(Q2)처럼 시간 계산 자체의 모호성이 남아있어 오히려 새로운 불일치를 만들 위험. A는 운영자 판단을 신뢰하는 가장 단순한 트리거로 이런 위험을 회피.

**영향**: PRD FR-28. `docs/PRD.md`에는 이미 FR-28로 반영되어 있으나 T4(3.4절) 사전조건 "총생산시간 경과"는 A 채택에 따라 "생산담당자의 수동 명령 실행"으로 해석해 구현.

**테스트를 위한 보강**:
- 생산큐 선두 PRODUCING 항목에 대해 "생산완료 처리" 명령 실행 → 해당 항목만 CONFIRMED로 전이되고 재고수량에 floor(실생산량×수율)만큼 반영되는지 검증.
- 생산큐가 비어있는 상태에서 "생산완료 처리" 명령 실행 → 처리할 대상이 없다는 명시적 안내(조용한 무시 금지, R5와 동일 원칙) 검증.
- FIFO 순서(Q11)상 큐 두 번째 이후 항목에는 영향이 없는지(선두 항목만 전이) 검증.

## R7. 시료ID 참조 무결성 검증 (RISKS §7 → PRD FR-29)

**결정**: 시료예약 시 입력된 시료ID가 실제 등록된 시료인지 검증하고, 시료등록 시 시료ID 중복 여부를 검증한다. 위반 시 처리를 거부하고 명확한 에러 메시지를 표시한다.

**선택안 (웹 리서치)**:
- **선택안 A — 앱 레벨 참조 존재 검증(FK류 체크)**: 삽입/수정 전 컬렉션/딕셔너리에서 참조 대상 존재 여부를 확인. 이 프로젝트처럼 별도 RDB가 없는 구조에서도 구현 가능.
- **선택안 B — DB 레벨 UNIQUE/FK 제약**: 관계형 DB의 고유키·외래키 제약으로 원자적 보장. 이 프로젝트는 파일 기반 영속성이라 직접 적용 불가.
- **선택안 C — idempotency key**: 클라이언트가 발급한 고유키로 중복 처리 방지. 동시 다중 클라이언트 환경에 적합하나 이 프로젝트(Q10, 단일 프로세스 순차 처리 전제)에는 불필요.
- **채택**: 선택안 A. B는 DB가 없어 적용 불가, C는 Q10 전제(단일 프로세스 순차 처리)상 과잉 대응이므로 배제.

**근거**:
- Q12(수치 속성 하한 검증)와 동일한 원칙("등록/입력 시점 검증")을 참조 무결성에도 일관 적용하는 것이 자연스러움. SPEC_ANALYSIS에서 이미 지적된 사각지대(존재하지 않는 시료ID로 주문, 중복 시료ID 등록)를 방치하면 정상 흐름 테스트에서는 드러나지 않다가 엣지케이스 QA에서 바로 노출되는 전형적 취약점.
- 딕셔너리/컬렉션의 키 존재 확인만으로 앱 레벨 검증이 충분히 구현 가능해 별도 인프라 도입 없이 저비용으로 해결 가능.

**영향**: PRD FR-29. FR-06(시료예약), FR-01(시료등록) 구현에 각각 참조 존재 검증/중복 검증 로직 추가.

**테스트를 위한 보강**:
- 등록되지 않은 시료ID로 시료예약 시도 → 거부되고 "시료ID를 찾을 수 없습니다" 메시지 반환, 주문 미생성 검증.
- 이미 등록된 시료ID로 시료등록 재시도 → 거부되고 "이미 등록된 시료ID입니다" 메시지 반환, 기존 시료 데이터 불변 검증.
- 정상 시료ID로 시료예약 시 정상 처리되는 회귀(기존 FR-06 테스트와 충돌 없음) 확인.

## R8. 영속성 저장 실패 복구 전략 (RISKS §8 → PRD FR-30)

**결정**: 저장은 임시 파일 작성 후 원자적 교체(atomic rename) 방식을 사용하고, 로드 시 파일 손상이 감지되면 백업본으로 폴백하거나 명확한 경고와 함께 새로 시작한다.

**선택안 (웹 리서치)**:
- **선택안 A — 임시파일 작성 후 원자적 교체(+백업 보존)**: 같은 볼륨 내 임시 파일에 먼저 쓰고 flush 후 rename으로 교체. 교체 시 직전 정상본을 백업 경로로 보존(`.bak`). 저장 도중 프로세스가 죽어도 원본 파일은 손상되지 않음.
- **선택안 B — 백업 + 버전/체크섬**: `data.json.bak` 유지, 스키마 버전 필드와 체크섬(SHA-256/CRC32)으로 손상(잘림 등) 탐지.
- **선택안 C — PID 기반 lock/WAL 결합**: 저장 실패 복구를 위한 WAL 도입. 이 규모 콘솔 앱에는 과설계.
- **채택**: 선택안 A + B 결합(임시파일 원자적 교체 + 백업본 보존, 필요 시 체크섬). C는 배제.

**근거**:
- 실제 현장 "영속성 버그"의 다수는 정상 저장/로드 경로가 아니라 비정상 종료 이후의 복구 경로에서 발생. 이 시스템은 Q15에서 자동증가 정수 ID를 채택했는데, 파일 손상으로 마지막 저장 시점을 알 수 없게 되면 다음 주문ID가 기존 주문과 충돌(중복 채번)할 위험이 있음.
- 임시파일 원자적 교체는 별도 DB 없이도 "쓰기 도중 크래시 시 원본 보존"이라는 핵심 속성을 저비용으로 얻을 수 있는 가장 실용적인 방식.

**영향**: PRD FR-30, FR-23(데이터 영속성)의 구체적 구현 방식으로 반영. Repository 구현체(§2 아키텍처 규칙 — Model/Controller는 Repository 인터페이스에만 의존)의 저장/로드 로직에 적용.

**테스트를 위한 보강**:
- 저장 도중 강제 종료를 시뮬레이션(예: 임시파일 작성 단계에서 프로세스 중단) → 재실행 시 직전 정상 데이터가 그대로 유지되는지 검증.
- 저장 파일을 인위적으로 손상(예: 파일 일부를 잘라내거나 깨진 내용으로 치환)시킨 후 재실행 → 백업본 폴백 또는 "저장 파일이 손상되어 새로 시작합니다" 경고가 표시되는지 검증.
- 정상 저장/로드 사이클(FR-23 기존 테스트)이 이 변경 이후에도 회귀 없이 통과하는지 확인.

## R9. 시료검색의 숫자 필드 일치 정책 확정 (RISKS §9 → PRD FR-31)

**결정**: 시료검색(FR-05) 시 수율·평균생산시간과 같은 숫자 속성은 정확 일치를 기본 정책으로 채택한다. 범위 검색은 이번 버전 범위에서 제외한다.

**선택안 (웹 리서치)**:
- **선택안 A — 정확 일치를 기본값**: 숫자 필드도 문자열 필드와 동일하게 정확 동등 비교로 매칭. 일반적인 CRUD/관리자용 검색 UI 다수가 채택하는 저비용 방식.
- **선택안 B — 비교/범위 연산자 문법**: `field:>100`, `field:10..50` 같은 연산자를 통합 검색창에 지원(GitHub 검색 문법, Jira JQL류).
- **선택안 C — 대괄호/구간 표기**: `[min max]`/`(min max)` 구간 표기(Lucene/OpenSearch류). 보통 UI 필터 뒤에 숨겨지는 방식이라 통합검색창 직접 타이핑에는 부적합.
- **채택**: 선택안 A(정확 일치 기본값). B/C는 이번 콘솔 앱 규모 대비 과한 문법 설계이므로 배제하되, README/PRD에 "숫자 필드는 정확 일치만 지원"함을 명시해 사용자 기대치를 관리.

**근거**:
- Q14(시료검색은 시료ID/이름/수율/평균생산시간 통합검색)가 숫자 속성의 정확 일치/범위 일치 여부를 "구현 자유"로 남겼는데, 이는 자유도가 아니라 미해결 모호성에 가깝다 — 채점자가 "수율로 검색해보세요"라고 요청했을 때 기대와 다른 결과가 나오면 "검색이 안 된다"는 인상을 줄 수 있음.
- 정확 일치는 구현 비용이 가장 낮고, 향후 범위 검색이 필요해지면 별도 필터 명령으로 추가하기 쉬운 확장 여지를 남긴다.

**영향**: PRD FR-31(Q14의 세부 정책 확정). FR-05 구현 시 수율/평균생산시간 검색어는 값의 정확 일치로만 매칭.

**테스트를 위한 보강**:
- 등록된 시료의 수율 값과 정확히 일치하는 검색어 입력 → 해당 시료가 결과에 포함되는지 검증.
- 근접하지만 일치하지 않는 값(예: 등록값 0.9에 대해 0.89 검색) → 결과에서 제외되는지 검증(범위 검색이 아님을 확인).
- 평균생산시간에 대해서도 동일한 정확 일치 케이스/제외 케이스 검증.

## R10. 감사 추적을 위한 타임스탬프 기록 (RISKS §10 → PRD FR-32)

**결정**: 각 주문의 생성 시각(CreatedAt)과 상태 전이 시각을 내부적으로 기록한다. 화면 표시 의무는 없다.

**선택안 (웹 리서치)**:
- **선택안 A — 감사 컬럼(CreatedAt/UpdatedAt류)**: 주문 등 영속화되는 엔티티에 생성/갱신 시각 필드를 기본으로 추가. 사실상 업계 기본값.
- **선택안 B — 상태전이 이력 로그(append-only, 이벤트소싱 라이트)**: 현재 상태만 저장하는 대신 각 전이를 불변 로그 행으로 기록. "현재 상태 + 이력 사이드 테이블" 조합.
- **채택**: 선택안 A(최소 CreatedAt) + 선택안 B의 경량 버전(전이 시각을 append-only 리스트로 함께 기록) 결합. 완전한 이벤트소싱(재생을 통한 상태 도출)은 이 규모에 과설계이므로 배제.

**근거**:
- 타임스탬프 없이 순서만 배열 인덱스(큐 위치)로 관리하면, 이 주문이 언제 접수됐는지·생산에 얼마나 걸렸는지를 사후에 검증할 방법이 없음. FIFO 생산큐(Q11) 순서, 대기주문확인(§5-5.2) 화면의 "언제부터 대기 중인지" 등을 보여주려면 최소한의 시각 정보가 필요.
- 이 필드는 초기에 추가해도 기존 비즈니스 로직에 영향이 없는 반면(additive), 주문이 필드 없이 쌓인 뒤에는 영구히 복구 불가능한 비대칭적 손실이 발생 — 지금 반영하는 것이 저비용.

**영향**: PRD FR-32. Order 엔티티(§2.2)에 SPEC/PRD에 명시되지 않은 내부 전용 필드(CreatedAt, 전이 이력)가 추가됨 — 화면 출력 의무는 없으므로 View 표시 규칙(§7 UI 자유)과 충돌하지 않음.

**테스트를 위한 보강**:
- 시료예약(주문 생성) 시 CreatedAt이 기록되는지 검증(값 자체의 정확성보다 "기록되었는지" 여부를 확인 — 시각 비교는 테스트 환경에 따라 불안정할 수 있으므로 "null이 아님/설정됨" 수준으로 검증).
- 각 상태 전이(T1~T6) 발생 시 전이 이력에 새 항목이 추가되는지 검증.
- 전이 이력이 실제 전이 순서(FIFO, Q11)와 일치하는 시간 순서로 쌓이는지 검증.

# CONTRACT 감사 기반 보강 결정 (C1~C3)

> **본 섹션은 위 Q1~Q16, R1~R10과도 근거 계통이 다르다.** Q1~Q16은 SPEC 자체의 모호점에 대한 결정, R1~R10은 RISKS.md 리스크 분석에서 도출된 결정인 반면,
> 아래 C1~C2는 `docs/CONTRACT.md`(v1)를 `docs/SPEC.md` §8 "미션1: PoC 개발" 산출물 요구사항 표와 대조 감사한 결과 발견된 공백 — "데이터영속성처리" 항목의 "CRUD 포함" 요구 중 **Delete(삭제) 연산 근거가 PRD/DECISIONS 어디에도 없던 문제** — 를 메우기 위한 결정이다.
> CONTRACT.md는 원래 PRD.md/DECISIONS.md만을 근거로 작성되었으나, PRD.md 자체가 SPEC §8을 별도로 검토하지 않아 이 요구가 누락되었다. 아래 결정은 그 근본 원인을 해소하고 PRD.md/CONTRACT.md에 소급 반영하기 위한 것이다.

## C1. 시료(Sample) 삭제 지원 여부 및 방식

**결정**: 시료 삭제(Delete)는 하드 삭제(레코드 물리적 제거)로 지원한다. 단, C2에서 정의하는 참조무결성 가드를 통과해야 한다. 별도의 소프트 삭제(비활성화 플래그 등)는 도입하지 않는다.

**근거**:
- 웹 리서치 결과, 일반적인 MRP/ERP의 품목 마스터(Item Master) 삭제 정책은 "참조하는 트랜잭션이 전혀 없을 때만 하드 삭제를 허용"하는 방식이 표준 관행이며, 참조가 있는 경우에는 비활성화(discontinue) 플래그로 전환하는 것이 일반적이다.
- 그러나 C2에서 채택하는 참조무결성 가드(해당 시료ID를 참조하는 Order가 하나라도 존재하면 삭제 자체를 거부)가 있으면, 한 번이라도 주문이 발생한 시료는 어차피 영구히 삭제될 수 없다. 즉 "참조 있음 → 비활성화"와 "참조 있음 → 삭제 거부"는 이 시스템에서 실질적으로 동일한 효과를 낸다.
- 이 상황에서 별도의 IsActive/Discontinued 필드와 그에 따른 조회·검색 로직(FR-04/FR-05) 분기를 추가로 두는 것은, 이미 반복된 "이 프로젝트 규모 대비 과설계 배제" 원칙(Q6, Q10, R2, R4 등)과 어긋난다. 하드 삭제 + 참조무결성 가드만으로 충분하다.
- 이 결정은 FR-29(R7, 시료ID 참조무결성 검증 — 주문 생성 시 "존재해야 함" 강제)와 대칭을 이루는 반대 방향 검증이다.

**영향**:
- PRD FR-33(신규, 시료 삭제)로 반영.
- CONTRACT.md §5 `ISampleRepository`에 `Delete` 메서드 추가.
- 삭제 가능 여부 판단(참조무결성 확인)은 Repository가 아니라 Model/Controller 계층의 책임이다(CLAUDE.md §2 아키텍처 규칙 — Repository는 저장 방식만 캡슐화, 비즈니스 규칙은 Model/Controller).

**이 결정이 위반되면 실패해야 할 테스트**: 어떤 Order도 참조하지 않는 등록된 시료ID에 대해 삭제를 실행했을 때, 시료가 삭제되지 않고 그대로 남아 있으면(또는 시료조회에 계속 나타나면) 테스트가 실패해야 한다.

## C2. 시료 삭제 시 참조무결성 정책

> **[Superseded by ADR-E7, ADR-E8 — 2026-07-15]** 아래 원문은 보존하되, 다음 두 지점이 뒤집혔다:
> (1) "호출자(Model/Controller)"로 병기했던 책임 주체는 ADR-E8에서 **Model 단독**으로 확정되었다.
> (2) "별도의 `FindBySampleId` 메서드를 Repository 인터페이스에 새로 추가하지 않고 기존 `FindAll()` 필터링으로 충분하다"는 판단은 ADR-E7에서 뒤집혀, `IOrderRepository::FindBySampleId`가 신설되었다.
> 결정의 이력을 보존하기 위해 원문은 그대로 둔다.

**결정**: 특정 시료ID를 참조하는 Order가 하나라도 존재하면(RESERVED/PRODUCING/CONFIRMED 등 진행 중 상태는 물론, REJECTED/RELEASE 같은 종단 상태의 과거 이력 포함 — 상태 무관) 해당 시료의 삭제 요청을 거부한다.

**근거**:
- 웹 리서치 결과, 관계형 DB의 `FK ON DELETE RESTRICT`나 SAP류 자재 마스터의 "미결/이력 문서(open or historical transactions)가 있으면 삭제 차단" 정책이 생산관리/재고관리 시스템의 표준 관행이다. 품목을 참조하는 트랜잭션 기록이 하나라도 있으면 마스터데이터 삭제를 막아, 과거 주문 조회 시 참조 무결성이 깨지는 것(예: 존재하지 않는 시료ID를 가리키는 Order)을 방지한다.
- REJECTED/RELEASE도 포함해 "상태 무관"으로 정한 이유: FR-32(R10)에서 이미 감사 추적을 위해 Order를 영구 보존하기로 결정했다. 종단 상태 Order라도 시료 참조가 끊기면 감사 기록의 무결성이 깨지므로, 감사 목적과 일관되게 이력 전체를 보호 대상으로 삼는다.
- 이 결정은 FR-29(R7)가 이미 확립한 원칙(참조 무결성은 애플리케이션 레벨에서 검증, 별도 RDB 제약 없이 컬렉션 존재 확인으로 충분)과 동일한 방식(선택안 A — 앱 레벨 검증)을 재사용한다. 새로운 인프라나 메커니즘을 도입하지 않는다.

**영향**:
- PRD FR-33(신규)의 사전조건으로 반영: "삭제 대상 시료ID를 참조하는 Order가 존재하지 않아야 한다."
- Model/Controller 계층에서 삭제 실행 전 `IOrderRepository::FindAll()` (또는 상태 필터 없이 전체 조회)로 참조 여부를 확인한다 — 이 규모에서는 별도의 `FindBySampleId` 메서드를 Repository 인터페이스에 새로 추가하지 않고 기존 `FindAll()` 필터링으로 충분하다(과설계 배제).
- 위반 시 FR-27(R5)과 동일한 원칙으로 "참조 중인 시료는 삭제할 수 없습니다" 등 명시적 거부 메시지를 표시한다(조용한 무시 금지).

**이 결정이 위반되면 실패해야 할 테스트**: 특정 시료ID를 참조하는 Order가 하나라도(REJECTED/RELEASE 이력 포함) 존재하는 상태에서 해당 시료 삭제를 시도했을 때, 삭제가 거부되지 않고 성공하거나 관련 Order가 존재하지 않는 시료ID를 참조하게 되면 테스트가 실패해야 한다.

## C3. Order 삭제 미지원 (감사 기록 보존 원칙과의 정합)

**결정**: Order는 어떤 상태에서도 하드 삭제를 지원하지 않는다. CONTRACT.md `IOrderRepository`에는 Delete/Remove류 메서드를 추가하지 않는다.

**근거**:
- 웹 리서치 결과, 실무 ERP/MES에서 오더(주문) 레코드는 컴플라이언스·감사 목적상 물리적으로 삭제하지 않고 상태 필드(취소/거절 등)로만 종결 처리하는 것이 표준 관행이다.
- 이 시스템에서 "주문을 없앤다"는 의도는 이미 Q6/Q7/Q8이 정의한 상태 전이(T3 거절, T6 CONFIRMED 취소)로 충분히 커버되며, REJECTED가 사실상의 논리적 삭제 역할을 한다.
- FR-32(R10)에서 Order에 CreatedAt·전이 이력을 기록하기로 결정한 목적 자체가 "추후 조회·디버깅을 위한 감사 추적"이다. Order를 하드 삭제 가능하게 하면 이 감사 추적 목적이 무의미해지므로, C1(시료 삭제 허용)과 달리 Order는 삭제를 아예 지원하지 않는 것이 R10과 정합적이다.

**영향**:
- PRD.md §6 비범위(Non-goals)에 "Order의 하드 삭제"를 추가.
- CONTRACT.md §5 `IOrderRepository`는 기존 그대로(Delete 메서드 없음)이며, 이는 누락이 아니라 의도적 설계임을 주석으로 명시한다.

**이 결정이 위반되면 실패해야 할 테스트**: `IOrderRepository`에 Order를 물리적으로 제거하는 메서드가 선언되어 있거나, 어떤 경로로든(직접 삭제 API, 파일 조작 등 정상 API 외 경로 제외) Order 레코드가 저장소에서 사라지면 테스트가 실패해야 한다.

# CONTRACT v3 오류 어휘 통합 및 감사 종결 결정 (E1~E11)

> **본 섹션은 `docs/AUDIT-CONTRACT-v2.md`(B-02 재감사)가 지적한 42개 실패 상황의 "닫힘"을 위한 결정이다.**
> 감사 기준 개정: "모든 실패 상황이 시그니처로 전달 가능해야 한다"는 과거 기준을 폐기하고,
> 닫힘 방식을 (a) 표현한다 / (b) 불가능하게 만든다 / (c) 범위 밖으로 선언한다(계약 명문화 + 근거 기록) 셋 중 하나로 재정의한다.
> 관용구 규칙도 "관용구 2개 이상 공존 금지" → "동일 범주 내 관용구 분화 금지"로 개정한다: 범주가 다르면 관용구가 달라도 되나,
> 범주 경계가 계약에 명문화되고 모든 메서드가 각각 어느 범주인지 표로 확정되어야 한다.

## E1. 실패 3범주 확정

**결정**: 저장소 계층의 모든 결과/실패를 다음 3범주로 나눈다. 각 범주의 관용구는 정확히 하나다.
- **[R] 정상 결과** — 부재는 실패가 아니다. 관용구: `optional<T>` / `vector<T>`.
- **[D] 도메인 거부** — 저장소는 정상이나 요청이 거부됨. 관용구: `[[nodiscard]] enum class WriteOutcome { Ok, NotFound, DuplicateKey }`.
- **[F] 치명** — 저장소를 신뢰할 수 없음. 관용구: 예외. 앱 경계에서 1회 포착 후 종료.

**근거**:
- AUDIT-CONTRACT-v2.md §4가 지적한 "관용구 4종(void/bool/optional/vector) + 정의되지 않은 5번째 채널(예외)" 공존 문제는, 관용구 개수 자체가 아니라 "어느 관용구가 어느 종류의 실패를 표현하는 책임을 지는지"가 계약에 없었던 것이 근본 원인이다. 범주를 먼저 확정하고 범주당 관용구 하나만 허용하면, 관용구가 여러 개(3개) 공존해도 각각의 책임 경계가 명확해져 감사가 지적한 "무엇을 보게 될지 알 수 없다"는 문제가 해소된다.
- `void`/`bool` 관용구(감사 #15~20, #21~25, #38~42)는 [D] 범주에 필요한 정보(무엇이 거부됐는지)를 표현할 수 없으므로 전량 폐기하고 `WriteOutcome`으로 교체한다.

**영향**: CONTRACT.md §5 전체 메서드 시그니처 개정(v3). Sample/Order의 Add/Update, Sample의 Delete가 `void`/`bool`에서 `WriteOutcome`으로 변경된다.

**이 결정이 위반되면 실패해야 할 테스트**: CONTRACT.md §5 표에 [R]/[D]/[F] 범주가 비어 있는 메서드가 하나라도 있으면, 계약 완결성 검사가 실패해야 한다.

## E2. 오류 어휘 6개 확정

**결정**: 저장 방식과 무관한 오류 어휘를 다음 6개로 확정한다.
- `WriteOutcome::Ok` / `NotFound` / `DuplicateKey`
- 예외: `StorageUnavailable`(저장소를 읽거나 쓸 수 없음) / `StorageCorrupted`(읽었으나 해석 불가 또는 불변식 위반) / `SchemaVersionMismatch`(저장된 스키마 버전이 프로그램이 아는 버전과 다름)

**근거**:
- AUDIT-CONTRACT-v2.md 감사축(대상 존재/부재, 키 중복, 신뢰 불가 데이터, 스키마 버전 다름, 저장 매체 접근 불가)을 그대로 어휘로 승격시키되, 구현 라이브러리(`std::ifstream`, `std::filesystem` 등)의 예외 타입이나 파일시스템 용어(`ENOENT`, `bad_alloc` 등)가 호출자에게 그대로 노출되면 안 된다 — Repository 구현체는 내부에서 발생한 모든 저장 계층 예외를 이 6개 어휘 중 하나로 반드시 재포장(wrap)해야 한다.

**영향**: CONTRACT.md에 6개 어휘를 §5의 일부로 명문화하고, 4개 PoC 저장소 구현체는 자신의 내부 예외를 이 어휘로 wrap해야 함을 계약에 명시한다. 예외 사양은 인터페이스 선언부 상단의 폐쇄 진술 1곳에 집중한다. 개별 시그니처 반복 표기는 하지 않는다 — 반복 표기는 나중에 하나가 누락될 때 그 누락 자체가 AUDIT-CONTRACT-v2.md §4가 지적한 "정의되지 않은 5번째 채널"을 재현하는 통로가 되기 때문이다.

**이 결정이 위반되면 실패해야 할 테스트**: 구현체가 저장 매체 접근 실패 시 6개 어휘 외의 예외 타입(예: `std::ios_base::failure`, `std::filesystem::filesystem_error`)을 호출자에게 그대로 전파하면 실패해야 한다.

## E3. Exists 제거

**결정**: `ISampleRepository::Exists`를 계약에서 제거한다. `FindById(id).has_value()`로 대체된다.

**근거**:
- AUDIT-CONTRACT-v2.md #11~14가 지적한 대로, `bool` 하나로는 [F] 범주(저장 매체 접근 불가, 데이터 신뢰 불가)를 표현할 수 없어 항상 `false`로 뭉개진다. `FindById`는 이미 같은 정보를 [R]/[F] 범주로 올바르게 분리해 제공하므로, `Exists`는 중복 인터페이스이면서 표현력이 더 낮은 열등한 대체재다.

**영향**: CONTRACT.md §5 `ISampleRepository`에서 `Exists` 선언 제거. 호출자(Model)는 `FindById(id).has_value()`를 사용한다.

**이 결정이 위반되면 실패해야 할 테스트**: `ISampleRepository`에 `Exists` 메서드가 여전히 선언되어 있으면 실패해야 한다.

## E4. NextOrderId 제거 (ADR-Q15는 유지)

> **경위**: 최초 초안은 "SPEC의 주문번호 채번 규칙이 일자별 시퀀스"라는 전제로 ADR-Q15(자동증가 정수 채번)를 뒤집으려 했으나, `docs/SPEC.md` 전문에 그런 규칙이 존재하지 않고 오히려 ADR-Q15 자체가 "SPEC에 채번 규칙 근거 없음"을 이미 기록하고 있어 모순이 발견되었다. 사용자 확인 결과, "일자별 시퀀스" 표기는 SPEC 예시 화면의 목업 문자열일 뿐 규범 요구가 아니며(ADR-Q15 보강 참고), ADR-Q15(자동증가 정수)는 그대로 유지하기로 확정했다.

**결정**: `IOrderRepository::NextOrderId`를 계약에서 제거한다. ADR-Q15(자동증가 정수 채번)는 유지된다. 채번은 도메인 서비스(Model)의 책임이며, Repository는 `Add`에서 `DuplicateKey`로 키 충돌만 거부한다.

**근거**:
- 채번 규칙은 방식(자동증가/일자별/그 외)과 무관하게 도메인 규칙이며 Repository의 책임이 아니다.
- 더 결정적으로, AUDIT-CONTRACT-v2.md #35·#37이 보인 것처럼 `int64_t` 단일 반환값 안에는 "이 값을 신뢰할 수 없다"를 표현할 자리가 없다. 저장 매체 접근 실패 시 임의값(0 또는 1)이 정상값으로 반환되어 이후 `Add`에서 키 충돌을 유발할 수 있다. 이 결함은 채번 방식과 무관하게 성립하므로, 채번 방식을 무엇으로 정하든 `NextOrderId`라는 형태의 메서드 자체가 근본적으로 결함이 있다.
- 제거 후 채번 경로: 도메인 서비스가 `FindAll()`로 기존 최대 orderId를 구해 +1 한다(저장소가 비어있으면 1). 저장 매체 실패는 `FindAll()`이 `StorageUnavailable`을 던져 표면화되므로, 임의값이 정상값으로 둔갑하는 경로가 소멸한다. `Add()`의 `DuplicateKey`가 최종 방어선이다.

**영향**: CONTRACT.md §5 `IOrderRepository`에서 `NextOrderId` 제거. AUDIT-CONTRACT-v2.md #34~#37은 (b) 불가능화로 닫힌다. 채번 로직은 Model(도메인 서비스)에 구현되며, 어떤 이유로도 `NextOrderId`류 메서드를 Repository 계약에 재도입하지 않는다.

**이 결정이 위반되면 실패해야 할 테스트**: `IOrderRepository`에 `NextOrderId`(또는 동등한 "다음 채번값을 반환하는" 메서드)가 다시 선언되어 있으면 실패해야 한다. 또한 저장 매체 접근 실패 상황에서 채번 로직이 임의의 정수를 정상값으로 반환하고 `Add`가 그것을 그대로 받아들이면(=`StorageUnavailable`이 표면화되지 않으면) 실패해야 한다.

## E5. Order 단일 키 — 기존 orderId 유지, 표시용 필드 신설 금지

**결정**: "표시용 주문번호 필드 없음" 분기를 택한다. `OrderRecord`에 새 필드를 신설하지 않는다. 기존 `orderId`(자동증가 정수, ADR-Q15)를 Order의 단일 키로 그대로 유지한다. 주문번호의 화면 표기는 View 계층의 관심사이며, 저장된 `orderId` 필드로부터 파생해서만 만든다. 표시용 문자열을 별도 필드로 저장하는 것을 금지한다.

**근거**:
- 표시용 문자열(예: `ORD-YYYYMMDD-NNNN`류)을 별도 필드로 저장하면 `orderId`와 표시번호라는 진실이 둘이 되고, 둘이 어긋나도(예: 마이그레이션 중 하나만 갱신) Repository 계약 수준에서 감지할 방법이 없다. E10의 불변식 절과 같은 이유로, 단일 진실 원천 원칙을 Order 키에도 적용한다.
- 표시 형식은 SPEC의 규범 요구가 아니므로(ADR-Q15 보강 참고) 계약이 지탱할 대상이 아니다. UI 자유(§7)에 맡긴다.

**영향**: CONTRACT.md §1.3 Order 필드는 v2와 동일(orderId 하나만 키). 화면에 특정 표기 형식(날짜 포함 등)이 필요하면 View가 `orderId`와 `createdAt`으로부터 순수하게 파생시켜 표시하고, 그 파생값을 저장하지 않는다.

**이 결정이 위반되면 실패해야 할 테스트**: `OrderRecord`에 `orderId` 외의 "표시용 주문번호"류 필드가 추가되어 있으면 실패해야 한다.

## E6. FindByStatus 인자를 열거형으로 교체

**결정**: `IOrderRepository::FindByStatus`의 인자를 `std::string`에서 `OrderStatus` 열거형으로 교체한다.

**근거**:
- CONTRACT.md §2가 이미 `Order.Status`의 5개 값을 열거값으로 정의하고 있음에도 시그니처가 문자열을 받는 것은 결함이다. AUDIT-CONTRACT-v2.md #31이 지적한 "§2.1 열거값 외 문자열 status 입력(오탈자 등)"은 문자열 인자이기 때문에 발생하는 문제이며, 열거형으로 바꾸면 오탈자라는 실패 상황 자체가 컴파일 타임에 불가능해진다.

**영향**: CONTRACT.md §2에 C++ `enum class OrderStatus { Reserved, Rejected, Producing, Confirmed, Release }`를 신설하고, 저장 시 문자열(§3 wire format)과의 매핑을 명시한다. `FindByStatus(OrderStatus status)`로 시그니처 변경. AUDIT-CONTRACT-v2.md #31은 (b) 불가능화로 닫힌다.

**이 결정이 위반되면 실패해야 할 테스트**: `FindByStatus`가 여전히 `std::string`을 인자로 받으면, 또는 열거값 외 문자열이 컴파일 없이 전달될 수 있으면 실패해야 한다.

## E7. IOrderRepository::FindBySampleId 신설

**결정**: `IOrderRepository::FindBySampleId(sampleId) -> vector<OrderRecord>`를 신설한다. `FindByStatus`와 동일한 형태([R] 범주)다.

**근거**:
- AUDIT-CONTRACT-v2.md §5가 지적한 대로, ADR-C2가 "참조무결성 검증(호출자)"을 요구하면서도 그 호출자가 무엇을 호출해서 확인해야 하는지 §5에 없었다. `FindAll()` 후 애플리케이션 레벨 필터링으로 대체 가능하다는 ADR-C2의 기존 판단은, 참조 대상 시료가 많아질수록 매번 전체 Order를 조회해 필터링해야 하는 비효율을 감수해야 하고, 무엇보다 "참조무결성 확인"이라는 반복적으로 필요한 연산에 전용 메서드가 없다는 계약상 공백을 그대로 남긴다. 전용 메서드를 신설하는 것이 계약을 명확하게 한다.
- 이 결정은 ADR-C2의 해당 판단을 뒤집는다(DECISIONS.md ADR-C2에 Superseded 표시 및 원문 보존).

**영향**: CONTRACT.md §5 `IOrderRepository`에 메서드 추가. ADR-E8(참조무결성 수임자)이 이 메서드를 호출한다. AUDIT-CONTRACT-v2.md §5(참조무결성 메서드 부재 지적)는 (a) 표현으로 닫힌다.

**이 결정이 위반되면 실패해야 할 테스트**: 참조가 있는 시료ID에 대해 `FindBySampleId`가 빈 vector를 반환하거나 메서드 자체가 존재하지 않으면 실패해야 한다.

## E8. 참조무결성 수임자 = Model 단독

**결정**: 참조무결성 검증의 수임자는 Model(도메인 서비스)이다. Controller가 아니다. Model은 `ISampleRepository::Delete` 호출 전에 `IOrderRepository::FindBySampleId(sampleId)`로 참조를 확인하고, 결과가 비어 있지 않으면 `Delete`를 호출하지 않는다. `ISampleRepository::Delete` 자체는 참조를 검사하지 않는다.

**근거**:
- CLAUDE.md §2 아키텍처 규칙("Controller는 조율만, 도메인 로직은 Model")에 따르면 참조무결성 검증은 도메인 로직이므로 Model이어야 한다. ADR-C2의 "호출자(Model/Controller)" 병기는 View/Controller가 교체되어도 불변식이 유지되어야 한다는 요구와 충돌한다 — Controller에 두면 다른 Controller 구현체로 교체 시 불변식이 소실될 수 있다.
- 이 결정은 ADR-C2의 해당 판단을 뒤집는다(DECISIONS.md ADR-C2에 Superseded 표시 및 원문 보존).

**영향**: CONTRACT.md에 "참조무결성" 절을 신설해 수임자(Model)와 그가 호출할 메서드(`IOrderRepository::FindBySampleId`)를 함께 명시한다.

**이 결정이 위반되면 실패해야 할 테스트**: Controller 계층 코드가 `IOrderRepository`를 직접 호출해 참조무결성을 검증하면(Model을 거치지 않으면) 실패해야 한다.

## E9. Repository는 도메인 규칙을 검증하지 않는다

**결정**: Repository가 검증하는 것은 키 유일성 / 대상 존재 / 저장 매체 상태뿐이다. Repository가 검증하지 않는 것은 참조무결성 / 상태 전이 적법성 / 수량 범위 / 그 밖의 모든 도메인 규칙이다.

**근거**:
- Repository가 도메인을 알면 저장 방식 교체 시 도메인 로직이 복제되어, "저장 방식을 교체해도 상위 계층은 바뀌지 않는다"는 계약의 근본 명제가 무너진다. AUDIT-CONTRACT-v2.md #39("Order.Add 시 존재하지 않는 sampleId 참조" 검증 주체 불명), #41("§4에 없는 불법 상태 전이로 Update 호출" 시 Repository가 순수 저장 계층인지 계약에 명문화 안 됨)은 이 원칙이 계약에 없어서 발생한 공백이다.

**영향**: CONTRACT.md §5에 "Repository 검증 범위" 절을 신설. #39, #41은 (c) 범위 밖 선언(Repository 계약 관점, 검증 주체는 Model — 기존 FR-29/FR-27과 동일 원칙)으로 닫힌다.

**이 결정이 위반되면 실패해야 할 테스트**: Repository 구현체가 존재하지 않는 sampleId를 참조하는 `OrderRecord::Add`를 거부하거나, §4에 없는 상태 전이를 담은 `Update`를 거부하면(도메인 규칙을 스스로 검증하면) 실패해야 한다 — 이는 Model의 책임을 Repository가 가로챈 것이다.

## E10. 불변식 2개 추가

**결정**: 계약에 다음 불변식을 추가한다.
- 모든 상태에 대한 `FindByStatus` 결과 크기의 합은 `FindAll` 결과의 크기와 같다.
- 저장된 상태(status) 값은 §2.1 열거값 중 하나여야 한다.
위반이 발견되면 `StorageCorrupted`로 취급한다.

**근거**:
- AUDIT-CONTRACT-v2.md #32("저장된 레코드의 status 필드가 열거값 외 문자열")가 지적한 "유령 주문"(FindAll에는 나타나지만 FindByStatus 어디에도 안 걸리는 레코드) 문제는, 이런 불변식이 계약에 없었기 때문에 발생한다. 불변식을 명문화하고 위반 시 어휘(StorageCorrupted)를 지정하면 유령 주문 상태 자체가 "감지 가능한 손상"으로 격상된다.

**영향**: CONTRACT.md에 "불변식" 절 신설. #32는 (a) 표현으로 닫힌다.

**이 결정이 위반되면 실패해야 할 테스트**: `FindByStatus` 5개 호출 결과 크기의 합이 `FindAll` 크기와 다르거나, 저장된 status 값이 §2.1 열거값 밖인데도 `StorageCorrupted`가 발생하지 않으면 실패해야 한다.

## E11. 비범위 4항목

**결정**: 다음을 계약의 비범위로 선언한다.
1. **스키마 마이그레이션**: 버전이 다르면 `SchemaVersionMismatch`를 던지고 종료한다. 변환하지 않는다.
2. **부분 손상 복구**: 레코드 하나가 깨지면 저장소 전체를 손상으로 간주한다. 손상 레코드를 건너뛰고 나머지를 반환하는 것을 금지한다.
3. **외부 조작으로 생긴 중복 레코드 탐지**: 정상 경로에서는 `DuplicateKey`로 차단된다. 저장 파일을 직접 편집해 생긴 중복은 (탐지된다면) `StorageCorrupted`로 취급하되, 탐지를 보장하지는 않는다.
4. **동시 쓰기**: 쓰기 프로세스는 하나로 가정한다. 모니터 도구는 읽기 전용이다.

**근거**:
- (1) SchemaVersionMismatch 시 변환 로직을 임의로 설계하면 검증되지 않은 마이그레이션 규칙을 시스템이 발명하게 된다(Q6/Q8과 동일한 scope 최소화 원칙).
- (2) 조용한 스킵은 "데이터가 원래 적었다"로 오인되어 은닉된 데이터 손실이 된다. AUDIT-CONTRACT-v2.md #8·#9가 이미 지적한 문제.
- (3) 정상 경로는 E1/E9(Add의 DuplicateKey)로 이미 닫혀 있다. 외부 파일 직접 편집으로 생긴 중복까지 완벽 탐지를 보장하는 것은 이 규모(ADR-Q10, 단일 프로세스 콘솔 앱)에 과잉설계다. AUDIT-CONTRACT-v2.md #2, #7, #23은 이 항목으로 닫힌다.
- (4) ADR-Q10(멀티 프로세스 동시 접근은 SPEC 범위 밖)과 동일한 원칙의 연장.

**영향**: CONTRACT.md에 "비범위" 절 신설. #2, #7, #23은 (c) 범위 밖 선언으로 닫힌다.

**이 결정이 위반되면 실패해야 할 테스트**: 스키마 버전이 다른 파일을 로드했을 때 자동 변환이 수행되면, 또는 손상된 레코드 하나를 건너뛰고 나머지가 정상 반환되면, 또는 두 프로세스의 동시 쓰기에 대한 방어 로직이 구현되어 있으면(과잉설계) 실패해야 한다.

# ADR-I 계열 — 통합 범위 결정 (Integration Scope)

> 본 섹션은 위 Q1~Q16/R1~R10/C1~C3/E1~E11과 근거 계통이 다르다. 앞선 계열들은 SPEC 자체의 모호점 해석(Q), 리스크 기반 보강(R), Repository 계약 감사(C/E)에 대한 결정인 반면, 아래 I1~I4는 **4개 PoC 저장소(ConsoleMVC/DataPersistence/DataMonitor/DummyDataGenerator)와 본 프로젝트(SampleOrderSystem) 사이의 통합 범위·방식**에 대한 결정이다. 이는 소급 기록이 아니라 2026-07-15 이 세션에서 사용자가 내린 신규 결정이다.

## I1. PoC 산출물의 이식 대상 지위 — repo 토폴로지 확정

**결정**: PoC 4종(ConsoleMVC, DataPersistence, DataMonitor, DummyDataGenerator)은 폐기 대상이 아니다. Walking Skeleton으로서 본 프로젝트(SampleOrderSystem)로 이식된다.

**맥락 (repo 토폴로지)**: `docs/CONTRACT.md`는 이미 4개 PoC 저장소 배포를 전제하고 있다. CONTRACT.md §CHANGELOG(278~282행)의 "영향받는 PoC 저장소" 열은 v1·v2·v3 모두 "ConsoleMVC, DataPersistence, DataMonitor, DummyDataGenerator (전체 4개)"로 일관 기록되어 있다. 다만 이는 CONTRACT.md가 계약 배포 대상 저장소를 전제한다는 것일 뿐, PoC의 실제 코드가 본 프로젝트(SampleOrderSystem)로 이식된다는 것과는 다른 명제다 — 이 절은 후자(이식 대상 지위)를 명문화한다. 이 절이 없으면 아래 I2~I4가 참조할 대상이 없어 읽히지 않는다. 산출물은 Public repo 5개로 구성된다:
- ConsoleMVC / DataPersistence / DataMonitor / DummyDataGenerator (PoC 4종, 각 개별 repo)
- SampleOrderSystem (본 프로젝트, 이 저장소)

CONTRACT.md는 SampleOrderSystem에서만 개정되며(CONTRACT.md 자체 명문화), PoC repo에 배포된 사본은 읽기 전용이다.

**근거**: `docs/SPEC.md` §8 "미션1: PoC 개발"이 "각 PoC는 개별 Repository로 생성하여 제출"을 요구하고 있고(SPEC.md 268~277행), CONTRACT.md는 이미 4개 PoC 저장소 존재를 전제로 작성되어 있다(v1 CHANGELOG부터 "영향받는 PoC 저장소: 전체 4개"로 일관 기록). 즉 PoC는 처음부터 일회성 검증물이 아니라 계약을 배포받는 대상으로 설계되어 있었다 — "버리지 않는다"는 이 설계 전제를 명문화하는 것이다.

**결과**: PoC 4종은 산출물 목록에서 제외되지 않는다. 통합은 PoC의 폐기가 아니라 PoC → 본 프로젝트로의 이식 작업이다.

**이 결정이 무효화하는 것**: 없음(신규 결정이며 기존 ADR을 뒤집지 않음). PoC를 통합 과정에서 삭제하거나 저장소 자체를 폐쇄하는 어떤 향후 결정도 이 ADR과 먼저 충돌 검토를 거쳐야 한다.

**결정 시점과 출처**: 2026-07-15 사용자 지시.

**기존 ADR과의 충돌 확인**: 없음. CONTRACT.md의 "4개 PoC 저장소에 동일 사본 배포" 전제와 정합적이며 이를 뒤집지 않는다.

## I2. 이식 방식 — v3 기준 재작성 (복사 후 수정 금지)

**결정**: PoC의 v1 코드를 복사한 뒤 고치는 방식을 쓰지 않는다. 이식은 CONTRACT.md v3 기준으로 코드를 처음부터 다시 작성하는 방식으로 한다.

**근거**:
- §2 (3) 확인 결과, 본 repo(SampleOrderSystem)의 커밋 14개 중 코드 커밋은 `d30ef7a Initial console app scaffold` 1건뿐이며, 저장소 루트에 `src` 디렉터리는 없고 `main.cpp` 스켈레톤만 존재한다(도메인 로직 없음). 기존 스캐폴드 `main.cpp`를 통합 시 어떻게 처리할지(유지/대체 등)는 이 ADR의 결정 사항이 아니며 별도 결정이 필요하다.
- 이 프로젝트는 "문서가 코드보다 먼저 커밋된다"는 것을 전제로 운영되어 왔다(CLAUDE.md 서두, 커밋 이력 전체가 그 증거). PoC의 v1 코드는 CONTRACT.md v1/v2 시점 계약을 대상으로 작성되었고 이후 v3까지 개정되었으므로, v1 코드를 그대로 복사해 들여오면 계약 위반 상태의 코드가 본 저장소 이력에 한 번이라도 커밋되는 순간, "문서가 코드보다 먼저"라는 전제가 이 저장소의 커밋 이력에서 깨진다.
- v1→v3 사이의 drift를 발견하고 고친 이력은 이미 각 PoC repo(ConsoleMVC 등)에 남아 있다. 여기서 그 수정 과정을 반복할 필요가 없다 — 이 저장소는 v3가 확정된 이후의 결과물만 받으면 된다.

**포기하는 것**: 재작성에는 비용이 든다 — PoC에서 이미 검증된 로직을 그대로 옮기지 못하고 다시 작성해야 하며, 원본 PoC 코드와 이식 결과 사이의 대응관계가 diff로 추적되지 않는다(복사+수정 방식이었다면 diff가 "무엇이 바뀌었는지"를 그대로 보여주지만, 재작성은 그 이력을 남기지 않는다). 이는 재작성 비용을 상쇄할 만큼 계약 정합성이 이 프로젝트에서 우선순위가 높다는 판단에 따른 선택이다.

**결과**: 이식 작업은 v1 코드 diff 기반이 아니라 CONTRACT.md v3 인터페이스 명세를 1차 사료로 하는 신규 구현으로 진행한다.

**이 결정이 무효화하는 것**: "PoC 코드를 그대로 복사해 온다"는 접근(직전 세션에서 논의되었으나 승인되지 않았던 계획)을 무효화한다. 그런 계획은 실행하지 않는다.

**결정 시점과 출처**: 2026-07-15 사용자 지시.

**기존 ADR과의 충돌 확인**: 없음.

## I3. DataMonitor — 프로세스 아키텍처 통합 제외 (기능 제외 아님)

**결정**: DataMonitor의 프로세스 아키텍처(별도 프로세스가 동일 저장 파일을 동시에 읽는 구조)는 본 프로젝트에 이식하지 않는다.

**근거 (ADR-Q10 원문 인용)**: `docs/DECISIONS.md` Q10 "결정": "단일 프로세스 내 순차 처리로 해결한다. 멀티 프로세스/멀티유저 동시 접근(예: 프로그램을 두 번 동시에 실행해 같은 저장소에 동시에 쓰기)은 SPEC 범위 밖(known limitation)으로 명시하고 별도 방어 로직(분산 락 등)은 구현하지 않는다." Q10 "영향": "README/설계 문서에 '본 시스템은 단일 프로세스 실행을 전제로 하며, 동일 데이터 저장소에 대한 멀티 프로세스 동시 접근은 지원 범위에서 제외한다'는 제약사항을 명시한다." DataMonitor는 그 정의상(별도 모니터링 프로세스가 동일 저장소를 동시에 읽는 구조) 이 범위 밖 전제 위에 서 있다.

**반드시 명문화할 사항**:
- **"통합 제외"는 "repo 폐기"가 아니다.** DataMonitor는 I1이 정의한 산출물 Public repo 5개 중 하나로 그대로 유지된다. 코드를 지우지 않는다.
- 제외되는 것은 **프로세스 아키텍처**(별도 프로세스가 파일을 동시에 읽는 구조)이지 **기능**이 아니다. FR-21(주문량확인) / FR-22(재고량확인)는 살아 있으며, 본 프로젝트 내에서는 단일 프로세스 내 Model 계산 + View 표시로 제공된다. 프로세스 분리 여부와 기능 존속 여부를 흐리지 않는다.
- 이것은 PoC의 실패가 아니라 정상 산출이다. DataMonitor라는 PoC가 자신의 설계 전제(별도 프로세스 동시 읽기)와 ADR-Q10(멀티 프로세스 동시 접근은 범위 밖)의 충돌을 스스로 발견한 것은 negative result이며, PoC가 원래 하도록 되어 있는 역할(통합 전에 이런 충돌을 값싸게 발견하는 것)을 정확히 수행한 것이다.
- **ADR-Q10은 이 결정으로 신규 채택되는 것이 아니다.** Q10은 이미 존재하는 기존 결정이며, I3은 Q10을 인용해 DataMonitor의 통합 범위를 정하는 것일 뿐 Q10 자체를 새로 만들거나 바꾸지 않는다. 이는 CONTRACT.md v4를 촉발하는 사유가 아니다.

**결과**: 본 프로젝트의 MVC 구현에는 별도 모니터링 프로세스를 두지 않는다. FR-21/FR-22는 단일 프로세스 내 Controller→Model 조회 경로로 구현한다.

**이 결정이 무효화하는 것**: 없음. Q10을 대체하지 않으며, DataMonitor repo나 그 코드를 무효화하지 않는다.

**결정 시점과 출처**: 2026-07-15 사용자 지시.

## I4. 테스트 프레임워크 미도입 — assert-count 방식 채택

**결정**: Catch2/GoogleTest/doctest 등 외부 테스트 프레임워크를 도입하지 않는다. 프레임워크 없는 assert-count 방식(assert 실행 결과를 직접 집계하는 자체 구현)으로 테스트를 구성한다.

**절차 근거**: `CLAUDE.md` §1은 "테스트 프레임워크(예: Catch2, doctest) 도입 여부와 테스트 프로젝트 구조를 먼저 사용자에게 확인한 뒤 구성한다 — 임의로 프레임워크를 도입하지 않는다"고 명시하고 있다. 이 확인이 2026-07-15 이 세션에서 이루어졌고, 그 답이 본 결정이다.

**실질 근거**:
- 외부 의존성 0. 빌드가 `msbuild SampleOrderSystem.vcxproj` 단독으로 재현 가능하며(CLAUDE.md §1 기존 빌드 명령), 별도 패키지 취득/버전 고정 문제가 발생하지 않는다.
- CLAUDE.md가 언급하는 Phase 6의 스냅샷 비교 로직("화면 문자열 비교 금지 / 저장 데이터 스냅샷 비교")은 어차피 자체 구현이 필요하다 — 프레임워크가 대신 해 주지 않는다. 프레임워크가 실제로 제공하는 것은 assert 집계·리포팅·필터링이며, 이는 이 프로젝트의 검증 축(저장 데이터 스냅샷 비교)과 직교한다.

**포기하는 것**: 프레임워크가 제공하는 리포팅(실패 위치의 서식화된 출력)·필터링(태그/이름으로 일부 테스트만 실행)·병렬 실행·IDE 테스트 러너 통합을 포기한다. assert-count 방식은 이런 편의를 스스로 구현하지 않는 한 갖지 못한다 — "그냥 안 썼다"가 아니라 이 편의들을 명시적으로 포기하고 빌드 재현성과 의존성 최소화를 택한 것이다.

**결과**: 테스트 harness는 외부 프레임워크 없이 assert 결과 집계(성공/실패 카운트) 방식으로 구성한다. 구체적인 harness 구조(파일 배치, 실행 방식)는 별도 설계가 필요하며 이 ADR의 범위가 아니다.

**이 결정이 무효화하는 것**: 없음(신규 결정).

**결정 시점과 출처**: 2026-07-15 사용자 지시.

**기존 ADR과의 충돌 확인**: 없음.
