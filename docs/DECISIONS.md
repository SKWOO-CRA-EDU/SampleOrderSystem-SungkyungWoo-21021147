# DECISIONS

> 본 문서는 `docs/SPEC_ANALYSIS.md`의 OPEN_QUESTIONS(Q1~Q16)에 대해, 개발팀이 실제로 채택한 결정을 기록한다.
> SPEC에 명시되지 않은 사항은 웹 검색을 통한 통상적인 생산관리/재고관리(MRP 등) 업계 관행을 참고하여 보완하며, 각 결정에는 근거를 남긴다.

## Q1. "부족분"의 계산식

**결정**: 부족분 = max(0, 주문수 - 재고수량)

**근거**:
- SPEC 어디에도 "부족분"의 계산식이 명시되어 있지 않음(SPEC_ANALYSIS.md Q1).
- 일반적인 MRP(자재소요계획, Material Requirements Planning)의 표준 공식은 다음과 같음:
  - Net Requirement = Gross Requirement − (Scheduled Receipts + On Hand Inventory)
  - 결과가 음수가 되는 경우(재고가 충분한 경우) 0으로 처리(max(0, ...))하는 것이 통상적인 관행.
  - 참고: [Gross to Net MRP — The Formula Explained](https://craftybase.com/blog/gross-to-net-calculations-for-mrp/), [SAP MRP Step 6 - Calculation of Shortage Quantities](https://sites.google.com/site/sapswords/home/sap-mrp/sap-mrp-functionality/step-6---calculation-of-shortage-quantities-and-creation-of-procurement-proposals-in-sap-mrp)
- 이 시스템에는 "입고 예정량(Scheduled Receipts)" 개념이 SPEC에 없으므로, 단순화하여 `부족분 = max(0, 주문수 - 재고수량)`으로 채택.
- max(0, ...)로 하한을 고정함으로써, 재고가 주문수보다 많은 경우(SPEC_ANALYSIS.md §3.2 "음수" 항목)에도 실생산량 = ceil(0/수율) = 0으로 자연스럽게 처리되어 별도 분기 로직이 불필요해짐.

**영향**:
- §3.2 실생산량 = ceil(부족분/수율) 계산의 입력값이 확정됨.
- Q5(여유/부족/고갈 경계)와 연동: 부족분 > 0이면 "부족", 부족분 = 0이면서 재고수량 = 0이 아니면 "여유"로 판정 가능해짐(단, Q5는 별도 결정 필요).

## Q2. §4 검증 절 수식값 vs 화면표기값 불일치 (실생산량 185 vs 206, 총생산시간 148 vs 165)

**결정**: 수식값(실생산량=185, 총생산시간=148)을 정답으로 채택한다.

**근거**:
- SPEC이 명시한 계산 공식(실생산량 = ceil(부족분/수율), 총생산시간 = 평균생산시간 × 실생산량) 자체는 §4·§5·§6에서 반복적으로 명확히 서술되어 있음.
- 반면 화면표기 예시값(206, 165)은 §4 검증 절에서 SPEC 스스로 불일치를 인정한 지점이며, 어떤 보정 로직에서 유래했는지 근거가 SPEC에 전혀 없음.
- 수식은 재현 가능하고 검증 가능하지만, 표기값은 출처 불명의 숫자이므로 수식을 정답으로 채택하는 것이 시스템의 일관성과 테스트 가능성 측면에서 안전함.

**영향**:
- 실생산량/총생산시간 계산 로직 및 관련 단위 테스트는 SPEC 수식을 기준으로 작성한다.
- SPEC 문서상의 화면표기 예시(206, 165)는 SPEC 자체의 오기로 간주하고, 구현체의 화면 출력값도 수식 계산 결과(185, 148 등)를 그대로 노출한다.

## Q3. 수율=0일 때 실생산량 = ceil(부족분/수율) 0으로 나누기 문제

**결정**: 수율=0(또는 0 이하)은 시료등록 시점에 유효성 검증으로 거부한다. 즉 런타임(생산 계산 시점)에서 0으로 나누기 상황 자체가 발생하지 않도록 입력 단계에서 차단한다.

**근거**:
- 웹 리서치 결과 제조업 수율(yield)은 통상 0%~100% 범위로 정의되며(Good Units / Total Units Started), 0%는 "전량 불량"을 의미하는 극단값으로 이론적으로는 존재 가능하나, 정상적인 생산관리 시스템에서는 데이터 유효 범위를 등록/입력 시점에 검증하여 이상값이 하류 계산에 전파되지 않도록 하는 것이 일반적 관행임.
- SPEC_ANALYSIS.md Q4(수율 허용범위 미정)와 근본적으로 동일한 문제이므로, Q4의 결정(수율 ∈ (0, 1])과 일관되게 처리하는 것이 합리적. 즉 수율=0은 애초에 저장이 불가능하므로 §3.2의 0으로 나누기 상황 자체가 시스템 내에서 발생하지 않는다.
- 런타임(생산 실행) 시점에 예외 처리를 추가하는 대안은 "이미 잘못된 데이터가 저장된 이후 매번 방어 코드로 걸러야 하는" 구조이므로, 입력 단계 차단보다 코드 복잡도와 오류 발생 지점이 늘어남.

**영향**:
- 시료등록(§5-1) 입력 검증 로직에 "수율은 0 초과 1 이하(0 < 수율 ≤ 1)여야 한다"는 규칙 추가 (Q4 결정과 연동).
- 실생산량 계산 로직에서는 수율=0 케이스에 대한 별도 방어 코드(예외 처리)가 불필요해짐 — 해당 상태 자체가 도달 불가능(unreachable)함을 전제로 구현.

## Q4. 수율의 허용 범위

**결정**: 수율 ∈ (0, 1] (0 초과 1 이하)

**근거**:
- SPEC 예시(90/100=0.9)는 비율(실수, 0~1 스케일)로 수율을 표현하고 있으며, 100%(수율=1, 불량 없음)는 물리적으로 가능한 값이므로 상한 포함.
- 0%(수율=0, 전량 불량)는 Q3 결정에 따라 등록 자체를 거부하므로 하한은 배제(초과 조건).
- 수율이 1을 초과(정상 시료 수가 총 생산시료 수보다 많음)하는 것은 정의상 모순이므로 상한을 1로 고정.

**영향**:
- Q3 결정과 동일한 검증 로직으로 구현: 시료등록 시 `0 < 수율 ≤ 1`이 아니면 등록을 거부(입력 오류로 처리).

## Q5. "여유/부족/고갈" 재고상태를 가르는 수치 경계

**결정**:
- 고갈: 재고수량 = 0
- 부족: 0 < 재고수량 < 주문수 (해당 시료의 판단 대상 주문수 기준)
- 여유: 재고수량 ≥ 주문수

**근거**:
- 업계 관행(재주문점/최소재고 등)은 절대 임계값을 쓰는 경우가 많으나, 이는 별도의 "최소재고" 같은 속성이 SPEC에 정의되어 있을 때 성립하는 방식이며, 본 SPEC에는 그런 속성이 없음.
- SPEC_ANALYSIS.md §3.4에서 이미 지적했듯, 이 판정이 "주문승인" 시의 재고충분/부족 판단(§3, §5-3)과 동일 기준을 공유하는지가 SPEC에 없는 문제였음 → 동일 기준을 공유하는 것으로 결정하여 Q1(부족분 = max(0, 주문수-재고수량))과 정합성을 맞춤: 부족분 > 0이면 "부족", 재고수량=0이면 "고갈"(부족분>0인 경우를 포괄하되 더 구체적인 케이스), 그 외(부족분=0이고 재고수량>0)면 "여유".
- 재고수량 자체에 대한 절대 기준(예: 5개 미만이면 부족)으로 정의할 경우, "부족" 상태가 특정 주문과 무관하게 존재해야 하는데 SPEC은 이런 절대 임계값을 어디에도 제시하지 않음. 반면 "주문수 대비"로 정의하면 SPEC이 유일하게 제공하는 수치(주문수)를 그대로 재사용할 수 있어 근거가 더 명확함.

**영향**:
- 모니터링(§5-4.2 재고량확인) 화면은 시료별로 해당 시료에 걸려 있는 활성 주문(RESERVED 등 판단 대상)의 주문수를 기준으로 여유/부족/고갈을 표기한다. 특정 주문과 무관한 일반 조회 시(예: 시료조회 §5-1.2)에는 이 판정을 적용하지 않고 재고수량만 노출한다.
- Q9(승인 시 재고 차감 시점)와 함께, "재고충분/부족" 판단 로직이 시스템 전체에서 일관된 단일 함수로 구현되어야 함.

## Q6. PRODUCING 상태 주문의 취소(거절) 가능 여부

**결정**: 미지원. PRODUCING → REJECTED 전이는 구현하지 않는다.

**근거**:
- 웹 리서치 결과, 실제 제조 시스템(ERP/MES)에서도 생산 중(In progress) 작업 지시를 취소하는 것은 "이미 투입된 자재·예약된 설비·큐 스케줄을 되돌리는" 복잡한 절차이며(예: SAP의 생산오더 취소 시 자재 소비 반제(reversal) 로직 필요), 단순한 상태값 변경 이상의 부수 처리가 필요함.
- 본 SPEC은 생산큐 항목 제거, 이미 소모된 자원(생산라인 점유 등)의 반환 규칙을 전혀 정의하지 않았으므로, 이런 되돌림 로직을 임의로 설계해 추가하면 SPEC에 없는 정합성 규칙을 시스템이 자체적으로 발명하게 되는 위험이 큼.
- 원칙적으로 SPEC에 명시되지 않은 기능은 구현하지 않는다(scope 최소화) — 미지원 쪽이 더 안전하고 이후 SPEC이 명확해지면 추가하기 쉬움.

**영향**:
- 상태 전이표(§2)에 PRODUCING → REJECTED 전이는 추가하지 않는다.
- 생산 중인 주문은 §3에서 정의된 대로 PRODUCING → CONFIRMED 전이만 가능하다.

## Q7. CONFIRMED(출고 대기중) 상태 주문의 출고 전 취소 가능 여부

**결정**: 허용. CONFIRMED → REJECTED 전이를 추가하며, 취소 시 승인 단계에서 차감된 재고를 복구한다.

**근거**:
- CONFIRMED는 생산이 이미 끝났거나(원래 재고충분) 필요 없던 상태로, PRODUCING과 달리 "이미 투입된 자재나 진행 중인 물리적 생산 공정"이 없어 취소에 따른 부수 효과가 재고 수치 복구 하나로 한정됨 → Q6(PRODUCING 취소, 미지원)과 달리 되돌림 로직이 단순함.
- 실무 주문관리 시스템에서도 "출고/배송 전 취소"는 일반적으로 지원되는 기능이며, 이 시점의 취소는 재고 예약 해제 정도로 처리 가능함(웹 리서치: shop order 취소 시 "reservations ... are also canceled, making the ... material reserved ... available for other shop orders").
- 단, 이 결정은 Q9(재고 차감 시점)의 결정에 의존한다: 재고가 승인 시점에 이미 차감되어 있어야 "복구"할 대상이 존재함.

**영향**:
- 상태 전이표(§2)에 CONFIRMED → REJECTED 전이 추가.
- 취소(거절) 처리 로직은 Q9에서 결정된 차감 시점 기준으로, 해당 주문의 주문수만큼 재고수량을 되돌린다.
- REJECTED는 여전히 최종 상태(더 이상 전이 없음)로 유지한다(Q8 참고).

## Q8. RELEASE가 최종 상태인지 (반품/재입고 지원 여부)

**결정**: RELEASE는 최종 상태로 확정한다. 반품/재입고 전이는 지원하지 않는다.

**근거**:
- Q6과 동일한 원칙: SPEC에 명시되지 않은 기능(반품, 재입고 시 재고/상태 처리 규칙)은 구현하지 않는다.
- RELEASE는 물리적으로 시료가 이미 고객에게 전달된 상태를 의미하므로, 반품 처리는 재고 복구뿐 아니라 품질 검사, 부분 반품 등 SPEC이 전혀 다루지 않는 추가 규칙이 필요해 scope를 크게 벗어남.

**영향**:
- 상태 전이표(§2)에서 RELEASE는 out-degree 0인 종단 상태로 확정한다.
- REJECTED, RELEASE 두 상태 모두 시스템상 최종(terminal) 상태로 취급한다.

## Q9. 재고 차감 시점 (주문승인 시 즉시 vs 출고 시 재확인)

**결정**: 주문승인(RESERVED→CONFIRMED 또는 RESERVED→PRODUCING) 시점에 재고수량에서 주문수를 즉시 차감한다.

**근거**:
- 승인 시점에 차감하지 않으면, 여러 주문이 동시에 "재고충분"으로 오판되어 실제 재고보다 많은 수량이 CONFIRMED 처리되는 오버부킹이 발생할 수 있음(Q10과 직결). 즉시 차감은 이 문제를 근본적으로 방지하는 가장 단순한 방법.
- Q7에서 결정한 "CONFIRMED→REJECTED 취소 시 재고 복구" 로직이 성립하려면, 애초에 승인 시점에 재고가 차감되어 있어야 복구할 대상이 존재함 → Q7 결정과 논리적으로 연결됨.
- SPEC §5-6의 "재고가 충분해진 CONFIRMED 주문을 출고 처리"라는 표현은, 승인 시점에 이미 재고가 차감(예약)된 상태에서 생산 완료로 재고가 채워져 "충분해지는" 것으로 해석 가능하며 모순되지 않음.

**영향**:
- 주문승인(§5-3) 로직: CONFIRMED/PRODUCING으로 전이하는 순간 재고수량 -= 주문수를 실행한다.
- 생산완료(PRODUCING→CONFIRMED) 시점에는 생산된 실생산량만큼 재고수량을 증가시킨다(§5-5/§6).
- 출고(CONFIRMED→RELEASE) 시점의 "재고가 충분해진" 조건은, 이미 예약 차감된 값이 생산 완료로 다시 채워졌는지를 확인하는 재검증으로 해석한다.

## Q10. 여러 주문이 동일 시료에 대해 동시에 승인 요청될 때의 순서/동시성 규칙

**결정**: 단일 프로세스 내 순차 처리로 해결한다. 멀티 프로세스/멀티유저 동시 접근(예: 프로그램을 두 번 동시에 실행해 같은 저장소에 동시에 쓰기)은 SPEC 범위 밖(known limitation)으로 명시하고 별도 방어 로직(분산 락 등)은 구현하지 않는다.

**근거**:
- SPEC §7 "영속성"은 "application을 다시 실행해도 데이터를 유지할 수 있는 성질"로 정의되어 있으며, 이는 재시작 간 데이터 보존을 의미할 뿐 멀티 프로세스/멀티유저 동시 접근을 요구하지 않음.
- 시스템은 콘솔 기반 단일 사용자 애플리케이션으로 설계되어 있어, 한 프로세스 안에서는 사용자가 한 번에 하나의 명령만 입력하므로 자연히 요청이 순차적으로 처리됨. 이 안에서는 Q9(승인 시 즉시 재고 차감)만으로 오버부킹이 방지됨.
- 동일 데이터 저장소에 대한 멀티 프로세스 동시 쓰기 방어(파일 락, DB 트랜잭션 격리 수준 등)는 SPEC이 전혀 요구하지 않는 별도의 인프라 수준 과제이며, 학생 프로젝트 성격의 콘솔 앱 범위를 벗어나는 과잉설계(overengineering)로 판단됨.

**영향**:
- 주문승인 로직은 별도의 락(lock) 없이, 요청이 들어온 순서대로 처리하는 것으로 구현한다(단일 프로세스 내 순차성 전제).
- README/설계 문서에 "본 시스템은 단일 프로세스 실행을 전제로 하며, 동일 데이터 저장소에 대한 멀티 프로세스 동시 접근은 지원 범위에서 제외한다"는 제약사항을 명시한다.

## Q11. 생산큐(FIFO)의 순차/병렬 생산 여부

**결정**: 순차 생산(단일 생산라인). 생산큐의 항목은 하나의 생산라인에서 한 번에 하나씩만 처리된다.

**근거**:
- SPEC에는 "생산라인"(단수)이라는 표현만 등장하고 라인 개수·병렬 capacity 개념이 전혀 없으므로, 병렬 생산을 지원하려면 SPEC에 없는 개념(라인 수, 자원 배정 규칙)을 임의로 발명해야 함.
- 단일 라인 순차 처리로 정하면 완료 시점 예측이 명확해짐: 특정 큐 항목의 예상 완료 시점 = 자신보다 앞선 모든 항목의 총생산시간 누적 + 자신의 총생산시간. 이는 "대기주문확인"(§5-5) 화면에서 대기 순서를 보여주는 목적과 자연스럽게 부합.
- 병렬 생산을 허용하면 "대기 중"이라는 개념 자체가 모호해짐(여러 항목이 동시에 진행 중이면 무엇이 "대기"인지 불명확).

**영향**:
- 생산큐는 FIFO 순서를 유지하는 단일 큐로 구현하고, 한 번에 하나의 항목만 PRODUCING 실행 상태로 처리한다.
- 대기주문확인(§5-5.2) 화면은 큐에 남은 항목과 그 순서를 그대로 노출하면 되며, 병렬 처리로 인한 순서 역전 문제를 고려할 필요가 없다.

## Q12. 수치 속성(주문수/재고수량/평균생산시간)의 하한/자료형 검증 규칙

**결정**: 등록/입력 시점에 검증한다.
- 주문수 > 0 (양의 정수)
- 재고수량 ≥ 0 (음이 아닌 정수)
- 평균생산시간 > 0 (양수)

**근거**:
- 웹 리서치 결과, 일반적인 주문관리 시스템의 검증 관행은 "자료형(숫자 여부) + 범위(주문수량에 허용 가능한 값) + 정책 정합성"을 주문/등록 시점에 자동 검증하는 것임(예: minimum order quantity 검증, order validation procedure에서 수량 필드 사전 검증).
- Q3/Q4에서 이미 확립한 원칙(수율=0/범위밖 값은 등록 시점에 거부)과 동일한 논리를 다른 수치 속성에도 일관되게 적용: 물리적으로 의미 없는 값(음수 주문량, 0 이하 생산시간)이 저장되면 이후 모든 계산식이 무의미한 값을 산출하는 문제(SPEC_ANALYSIS.md Q12에서 지적)를 입력 단계 검증으로 원천 차단.
- 재고수량은 0을 포함해야 함(§3.4 "고갈"이 재고수량=0으로 정의되므로 0은 유효한 정상 상태).

**영향**:
- 시료등록(§5-1.1) 입력 검증: 평균생산시간 > 0.
- 시료예약(§5-2) 입력 검증: 주문수 > 0(정수).
- 재고수량은 시스템 내부 로직(승인 시 차감, 생산완료 시 증가)에 의해서만 변경되며, 항상 0 이상을 유지하도록 하는 불변식을 코드 레벨에서 보장한다(Q1의 max(0, ...) 설계와 일관).

## Q13. 실생산량 중 재고에 반영되는 수량 (실생산량 전체 vs 실생산량×수율)

**결정**: 실생산량 × 수율(내림 또는 정수화 규칙은 별도 결정 필요)을 재고에 반영한다. 즉 "실생산량"은 "생산 시도 수(총 투입량)"를 의미하며, 이 중 수율만큼만 정상품으로 재고에 더해진다.

**근거**:
- 웹 리서치 결과, 제조업 표준 용어에서 "Total units started/produced"(생산 시도/투입 수량)와 "Good units produced"(정상품/양품 수량)는 명확히 구분되며, 완제품 재고(finished goods inventory)에 반영되는 것은 항상 후자(Good Units = Total Units × Yield)임.
- "실생산량 = ceil(부족분/수율)"이라는 SPEC의 수식 자체가 "수율 손실을 감안해 부족분만큼의 정상품을 확보하려면 얼마나 많은 양을 생산 시도해야 하는가"를 계산하는 역산 공식이므로, 실생산량은 "시도 횟수/투입량"의 의미로 봐야 정합적임. 만약 실생산량 전체를 그대로 재고에 반영하면 ceil과 수율 나눗셈 자체가 무의미해짐(수율이 얼마든 상관없이 항상 부족분 이상이 채워지므로 애초에 수율로 나눌 이유가 없어짐).
- ceil 연산으로 인해 실생산량×수율은 부족분보다 같거나 약간 큰 값이 되어(예: 부족분=100, 수율=0.9 → 실생산량=ceil(100/0.9)=112 → 정상품=112×0.9=100.8), 부족분을 충족하는 최소 시도 수를 계산하는 SPEC의 의도와 부합함.

**영향**:
- 생산완료(PRODUCING→CONFIRMED) 처리 시, 재고수량 += floor(실생산량 × 수율) (정수화 방식은 반올림 규칙이 SPEC에 없으므로 재고가 실제보다 부풀려지지 않도록 내림(floor)을 기본으로 채택 — 추가 논의 필요 시 별도 Q로 분리 가능).
- "실생산량"이라는 용어는 화면 표기 시 "생산 시도 수량"의 의미로 사용하고, 필요 시 별도로 "정상품 수량(=실생산량×수율)"을 함께 노출하는 것을 고려한다.

## Q14. 시료검색(§5-1.3) "속성으로 검색"의 대상 범위

**결정**: 시료ID, 이름, 수율, 평균생산시간을 모두 포함하는 통합검색으로 구현한다.

**근거**:
- SPEC이 "속성 기반 검색"이라고만 서술하고 특정 속성으로 한정하지 않았으므로, 시료 엔티티(§1.1)에 정의된 모든 속성(재고수량 제외 — 이는 파생/조회 전용 값이라 검색 키라기보다 결과 표시 항목에 가까움)을 검색 대상으로 포함하는 것이 SPEC의 "속성으로 검색"이라는 포괄적 문구에 가장 부합함.
- 이름만으로 한정하면 SPEC이 굳이 "이름검색"이라 하지 않고 "속성검색"이라고 표현한 이유를 설명하지 못함.

**영향**:
- 시료검색 기능은 입력된 검색어를 시료ID/이름(문자열 부분일치), 수율/평균생산시간(숫자 값 또는 범위 일치)에 대해 교차 검색하여 결과를 반환한다.
- 구체적인 숫자 속성(수율, 평균생산시간) 검색 시 정확 일치/범위 검색 여부 등 세부 UX는 구현 자유(§7 "UI 자유" 원칙)로 남긴다.

## Q15. 주문ID 채번 방식

**결정**: 자동증가 정수(1, 2, 3, ...)를 사용한다.

**근거**:
- 본 시스템은 단일 프로세스 콘솔 애플리케이션(Q10 결정 참고)이므로 분산 환경에서의 ID 충돌을 걱정할 필요가 없어 UUID의 이점(전역 고유성)이 불필요함.
- 자동증가 정수는 구현이 가장 단순하고, 콘솔에서 사용자가 "특정 주문"을 지칭할 때(§3, §5-3, §6) 입력하기도 쉬우며, 생성 순서를 그대로 보여주어 FIFO 생산큐 순서(Q11)와도 개념적으로 자연스럽게 연결됨.

**영향**:
- 주문 엔티티(§1.2)에 정수형 주문ID 속성을 추가하고, 시료예약(§5-2) 시 시스템이 자동으로 다음 순번을 부여한다.
- 콘솔 명령에서 "특정 주문"을 지정할 때는 이 정수 ID를 인자로 사용한다.

## Q16. 곱셈 기호 "×" 표기 적용 여부

**결정**: 화면 표기에 "×" 기호를 적용한다.

**근거**:
- SPEC §4 각주에서 명시적으로 "×"로 표기하라는 지시가 존재하며, 정식 UI 섹션(§5, §6)에 반복되지 않은 것은 SPEC 문서의 구조적 중복 누락(SPEC_ANALYSIS.md에서 지적된 §6의 중복 서술 패턴)일 뿐 지시 자체가 철회된 것은 아님.
- 명시적으로 존재하는 지시를 따르는 것이 안전하며, 화면 가독성 측면에서도 "×"가 "*"보다 일반 사용자에게 더 친숙한 표기임.

**영향**:
- 총생산시간 계산 결과를 화면에 표기할 때 "평균생산시간 × 실생산량 = 총생산시간" 형태로 출력한다.
