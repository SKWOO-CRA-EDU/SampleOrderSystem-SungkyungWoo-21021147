# SPEC_ANALYSIS

> 본 문서는 `docs/SPEC.md` 만을 입력으로 삼아 작성되었다. SPEC에 명시되지 않은 내용은 추측으로 채우지 않고 "미정/불명"으로 표기하거나 OPEN_QUESTIONS로 옮겼다.

## 1. 엔티티 후보와 속성, 불변식(invariant) 후보

### 1.1 Sample (시료)

SPEC §5-1 "시료등록"에서 도출.

| 속성 | 출처 | 비고 |
|---|---|---|
| 시료ID | §5-1 | 고유 식별자로 추정(SPEC에 "고유" 명시는 §5-1 서두 "각 시료는 고유한 이름과 속성을 가지며"뿐, ID의 고유성 자체는 명시 안 됨) |
| 이름 | §5-1 | - |
| 평균 생산시간 | §5-1 | 단위(분/시간 등) 미명시 |
| 수율 | §5-1 | §4 정의: 정상시료/총생산시료. 범위 미명시 |
| 재고수량 | §5-1 "시료조회" | 시료 자체 속성인지, 별도 재고 원장(ledger)에서 파생되는 값인지 SPEC에 구분 없음 |

**불변식 후보 (SPEC에 명시되지 않았으므로 어디까지나 "후보")**
- 재고수량 ≥ 0 — SPEC에 명시 없음
- 수율 ∈ (0, 1] — SPEC 예시(0.9)만 있고 범위 정의 없음 → Q4
- 평균생산시간 ≥ 0 — SPEC에 명시 없음
- 시료ID 유일성 — SPEC에 명시 없음

### 1.2 Order (주문)

SPEC §3, §5-2, §5-3에서 도출.

| 속성 | 출처 | 비고 |
|---|---|---|
| 시료ID | §5-2 "예약시 입력값" | 참조 무결성(존재하는 시료인지) 검증 언급 없음 |
| 고객명 | §5-2 | 자유 텍스트로 추정, 고객 엔티티와의 연결 여부 불명 |
| 주문수 | §5-2 | 하한/정수 여부 등 제약 없음 |
| 상태 | §3 | RESERVED/REJECTED/PRODUCING/CONFIRMED/RELEASE |
| 주문ID | 미명시 | "특정 주문"을 지칭하는 서술(§3, §5-3, §6)은 반복되나 식별자 자체는 SPEC 어디에도 정의되지 않음 → Q15 |
| 생성/전이 시각 | 미명시 | 모니터링·FIFO 정렬에 필요할 것으로 추정되나 SPEC에 속성으로 등장하지 않음 |

**불변식 후보**
- 주문수 > 0 — SPEC에 명시 없음
- 상태는 §3 표의 5개 값 중 하나 — SPEC 명시
- REJECTED는 "정상흐름 외 상태"이며 모니터링에서 제외 — SPEC §3 명시 (이는 불변식이라기보다 표시 규칙)

### 1.3 ProductionJob / 생산큐 항목 (생산라인 대기열 요소)

SPEC §5-5, §6 "생산"에서 도출. 별도 엔티티로 명시된 이름은 없고 "생산큐", "생산라인" 이라는 용어만 등장.

| 속성 | 출처 | 비고 |
|---|---|---|
| (참조) 주문 | §3, §5-5 | PRODUCING 상태의 주문과 1:1 대응으로 추정 |
| 부족분 | §3 "생산처리" | 정의식 자체가 SPEC에 없음(주문수-재고수량으로 역산 추정) → Q1 |
| 실생산량 | §4 | ceil(부족분/수율) |
| 총생산시간 | §4 | 평균생산시간 × 실생산량 |
| 큐 순서 | §5-5 "대기주문확인" | FIFO로 명시 |

**불변식 후보**
- 생산큐는 FIFO 순서를 유지한다 — SPEC 명시
- PRODUCING 상태의 주문은 정확히 하나의 생산큐 항목과 대응한다 — SPEC에 명시 없음(추정)

### 1.4 역할(Actor) — 엔티티인지 불명

§2에 "고객/주문담당자/생산담당자" 3종이 표로 정의되어 있으나, 이들이 시스템 내부에 저장되는 엔티티(계정, 로그인 등)인지, 단순히 업무 흐름 설명을 위한 역할 개념인지 SPEC은 구분하지 않는다. §7 "콘솔" 절에도 로그인/권한 개념이 등장하지 않는다.

## 2. 상태 전이표

상태 집합(§3): RESERVED, REJECTED, PRODUCING, CONFIRMED, RELEASE

| From | To | 트리거 | 사전조건(SPEC 근거) | 사후조건(SPEC 근거) |
|---|---|---|---|---|
| (신규) | RESERVED | 시료예약 (§5-2) | 시료ID가 시스템에 등록되어 있음("등록된 시료만 주문 가능", §5 서두) | 상태=RESERVED |
| RESERVED | CONFIRMED | 주문승인, 재고충분 (§3 "주문승인", §5-3) | 상태=RESERVED, "재고가 충분한 경우"(기준 수식 SPEC에 없음 → Q5) | 상태=CONFIRMED |
| RESERVED | PRODUCING | 주문승인, 재고부족 (§3 "주문승인", §5-3) | 상태=RESERVED, "재고가 부족한 경우" | 상태=PRODUCING, "생산라인에 자동으로 등록"(생산큐 추가) |
| RESERVED | REJECTED | 주문거절 (§3, §5-3) | 상태=RESERVED | 상태=REJECTED |
| PRODUCING | CONFIRMED | 생산완료 (§3 "생산처리", §5-5, §6) | 상태=PRODUCING, "생산완료시" (완료 판정 기준 SPEC에 없음) | 상태=CONFIRMED |
| CONFIRMED | RELEASE | 출고처리 (§3, §5-6) | 상태=CONFIRMED, "재고가 충분해진"(재고 재확인 기준 불명 → Q9) | 상태=RELEASE |

### 정의되지 않은 전이 (SPEC에 언급이 전혀 없음)

- REJECTED → (그 어떤 상태로도) : 거절 취소/복구 전이 없음
- RELEASE → (그 어떤 상태로도) : 출고 후 반품/재입고 등 전이 없음
- PRODUCING → REJECTED : 생산 중 주문 취소 가능 여부 불명 → Q6
- PRODUCING → RESERVED : 역행 전이 없음
- CONFIRMED → REJECTED : 출고 전 취소 가능 여부 불명 → Q7
- CONFIRMED → PRODUCING : 역행 전이 없음 (예: 승인 후 재고가 다시 부족해지는 경우)
- RESERVED → RESERVED : 예약 내용 수정(수량 변경 등) 전이 없음
- PRODUCING → PRODUCING : 부분 생산/재시도(수율 미달로 목표 미달성 시) 전이 없음

## 3. SPEC에 등장하는 모든 수식과 경계조건

SPEC §4, §5-1, §5-5/§6 "생산"에 등장하는 수식 전문.

### 3.1 수율 = 정상적인 시료 / 총 생산시료

- **0**: 총 생산시료 = 0 → 정의되지 않음(0으로 나눔). SPEC에 처리 방법 없음.
- **음수**: 정상적인 시료 또는 총 생산시료가 음수인 경우 SPEC에 언급 없음(물리적으로 불가능하다고 가정할 수 있으나 검증 규칙 없음).
- **나눗셈**: 정수/정수 나눗셈인지 실수 나눗셈인지 SPEC은 예시(90/100=0.9)로만 실수 나눗셈임을 암시.
- **반올림 방향**: 언급 없음(수율 자체는 반올림 대상으로 보이지 않음).
- **오버플로**: 총 생산시료가 매우 큰 정수일 경우 오버플로 가능성, SPEC에 자료형 언급 없음.
- **범위**: 수율이 0 또는 1을 초과할 수 있는지(불량 100%, 또는 정상 시료가 총생산시료를 초과하는 모순 데이터) SPEC에 검증 규칙 없음 → Q4.

### 3.2 실생산량 = ceil(부족분 / 수율)

- **부족분의 정의 자체가 SPEC에 없음.** "부족분"이라는 용어만 반복 사용되고(§3, §4, §5, §6) 계산식은 어디에도 명시되지 않음 → Q1.
- **0**: 부족분 = 0 → ceil(0/수율) = 0. 다만 이 케이스는 "재고 충분"으로 분류되어 애초에 이 수식이 호출되지 않을 상황과 경계가 겹침(부족분=0이 "충분"인지 "부족"인지의 경계 기준 자체가 Q5와 연결).
- **음수**: 부족분이 음수(재고가 주문수보다 많은 상태)일 경우 SPEC에 언급 없음.
- **나눗셈**: 수율=0이면 0으로 나눔 → 정의되지 않음(§3.1과 동일 문제가 전파됨).
- **반올림 방향**: ceil(올림)로 명시(SPEC §4, §5, §6에서 반복 확인됨). 이 부분은 SPEC이 명확한 몇 안 되는 지점.
- **정수 오버플로**: 부족분이 매우 크고 수율이 매우 작으면 실생산량이 극단적으로 커질 수 있음. 자료형(정수 범위) 명시 없음.
- **검증 불일치**: §4 "검증" 절 자체가 SPEC 문서 내에서 수식과 화면표기 예시값이 어긋남을 기록하고 있음(계산값 185 vs 표기값 206). SPEC이 스스로 모순을 인정하고 "미정"으로 남긴 유일한 지점 → Q2.

### 3.3 총생산시간 = 평균생산시간 × 실생산량

- **0**: 실생산량=0 → 총생산시간=0. 평균생산시간=0인 경우도 결과 0(현실적으로 비정상 값이나 SPEC에 검증 규칙 없음).
- **음수**: 두 입력 모두 음수가 안 된다는 명시적 규칙 없음.
- **나눗셈**: 해당 없음(곱셈 수식).
- **반올림 방향**: 언급 없음. 평균생산시간이 실수이고 실생산량이 정수이므로 결과가 실수가 될 수 있는데, 표시 시 반올림 여부/단위(분) 처리 규칙이 SPEC에 없음.
- **정수 오버플로**: 실생산량이 매우 크면 오버플로 가능, 자료형 언급 없음.
- **검증 불일치**: §4 검증 절에서 계산값 148 vs 표기값 165로 불일치 확인됨 → Q2와 동일 사안.
- **표기 기호**: 수식 자체는 "*"이나 UI 표기 시 "×"로 바꾸라는 지시가 §4 하단 각주에만 있고 정식 UI 섹션(§5, §6)에는 각주로만 반복됨 → Q16.

### 3.4 여유 / 부족 / 고갈 (재고상태 판정)

- SPEC(§4, §5-4)은 "여유: 재고충분", "부족: 재고수량부족", "고갈: 수량이 0"이라는 **정성적 서술만** 제공하며, 여유/부족을 가르는 **수치 경계(수식)가 SPEC에 없음**.
- **0 경계**: "고갈"은 정확히 재고수량=0으로 정의됨(SPEC에서 유일하게 숫자로 정의된 경계).
- **여유/부족 경계**: 재고수량 ≥ 주문수 이면 "여유"인지, 재고수량 > 0이면 무조건 "여유"이고 "부족"은 별도 조건인지 등 경계값이 SPEC에 없음 → Q5.
- 이 판정이 "주문승인" 시의 "재고 충분/부족" 판단(§3, §5-3)과 동일한 기준인지도 SPEC은 명시하지 않음(같은 임계값을 공유하는지, 별개 화면 전용 로직인지 불명).

## 4. 기능 메뉴 트리

SPEC §5 "UI 메뉴 트리 + 메뉴별 기능" (6개 메뉴, §0 포함 시 7개 항목) 기준.

```
0. 메인메뉴
   - 시스템 제목/소개, 시스템현황/시간/스펙, 메뉴 목록, 콘솔입력창

1. 시료 관리
   1.1 시료등록   (입력: 시료ID, 이름, 평균생산시간, 수율)
   1.2 시료조회   (등록 시료 목록 + 재고수량)
   1.3 시료검색   (목록 + 재고수량 + 속성 기반 검색; 검색 대상 속성 범위 불명 → Q14)

2. 시료 주문
   2.1 시료예약   (입력: 시료ID, 고객명, 주문수 → 상태 RESERVED)

3. 주문 승인/거절
   3.1 접수된 주문목록  (RESERVED 목록)
   3.2 주문승인         (재고충분 → CONFIRMED / 재고부족 → PRODUCING)
   3.3 주문거절         (→ REJECTED)

4. 모니터링
   4.1 주문량확인  (상태별 목록: RESERVED/CONFIRMED/PRODUCING/RELEASE, REJECTED 제외)
   4.2 재고량확인  (시료별 재고수량 + 여유/부족/고갈 표기)

5. 생산 라인
   5.1 생산현황표기  (현재 생산중 시료 정보; 표기 수준 SPEC상 "자율 결정"으로 명시 → Q(참고, 이는 SPEC이 스스로 미정임을 인정한 지점))
   5.2 대기주문확인  (생산큐, FIFO)
   (부족분 생산 → 실생산량/총생산시간 계산 → 생산완료시 PRODUCING → CONFIRMED)

6. 출고 처리
   (CONFIRMED 주문 → 출고 실행 → RELEASE)
```

§6 "생산 라인 규칙"은 §3/§5의 주문·생산·출고 규칙을 다시 한 번 반복 서술한 절로, 메뉴 트리 자체에 새 항목을 추가하지 않는다(내용 중복 → SPEC 자체 중복 구조, OPEN ITEMS 3번에서 이미 지적됨).

## 5. OPEN_QUESTIONS

Q1. "부족분"의 계산식이 SPEC 어디에도 정의되어 있지 않다(§3, §4, §5, §6 전부 용어만 사용). 이걸 잘못 정하면: 실생산량·총생산시간 계산 전체의 입력값 자체가 팀마다 달라져, 생산라인 시뮬레이션 결과가 서로 재현 불가능해진다.

Q2. §4 "검증" 절에서 실생산량(수식값 185 vs 화면표기 206), 총생산시간(수식값 148 vs 화면표기 165)이 불일치하며 SPEC 스스로 "미정"으로 남겨두었다. 이걸 잘못 정하면: 어느 쪽을 정답으로 채택하든 다른 쪽 기준으로 작성된 테스트/검수 자료와 충돌해 수용 기준(acceptance) 자체가 무효화된다.

Q3. 수율=0(전량 불량)일 때 실생산량 = ceil(부족분/수율) 계산이 0으로 나누기가 되어 정의되지 않는다. 이걸 잘못 정하면: 런타임 예외로 생산 로직이 죽거나, 무한대/NaN 값이 재고·주문 상태에 그대로 흘러들어가 시스템 전체가 오염된다.

Q4. 수율의 허용 범위(0 초과 1 이하 등)가 SPEC에 없다. 이걸 잘못 정하면: 시료등록 시 음수나 1 초과 수율이 그대로 저장되어, 이후 모든 생산량·생산시간 계산이 물리적으로 불가능한 값을 정상값처럼 산출한다.

Q5. "여유/부족/고갈" 재고상태를 가르는 수치 경계(부족분 임계값)가 SPEC에 없다("고갈=0"만 명시). 이걸 잘못 정하면: 모니터링 화면의 재고상태 표시가 실제 재고부족 여부와 불일치해, 담당자가 §1 배경에서 지적된 "재고 파악이 어렵다"는 원래 문제를 그대로 재현하게 된다.

Q6. PRODUCING 상태의 주문을 취소(거절)할 수 있는 전이가 SPEC에 정의되어 있지 않다. 이걸 잘못 정하면: 이미 생산 중인 불필요한 주문을 취소할 방법이 없어 생산라인 자원이 낭비되거나, 임의로 취소 로직을 추가할 경우 생산큐 정합성(이미 큐에 들어간 항목 처리)이 깨진다.

Q7. CONFIRMED 상태(출고 대기중) 주문을 출고 전에 취소할 수 있는 전이가 SPEC에 없다. 이걸 잘못 정하면: 확정 후 취소 요청을 처리할 방법이 없거나, 임의 구현 시 재고 복구(승인 시 차감된 재고를 되돌리는) 로직 누락으로 재고수량이 실제와 어긋난다.

Q8. RELEASE가 최종 상태로서 더 이상 전이가 없는지 SPEC이 명시적으로 서술하지 않는다(단지 다른 전이가 안 나올 뿐). 이걸 잘못 정하면: 출고 후 반품/재입고 시나리오의 지원 여부가 팀마다 갈려 데이터 무결성 검증 기준이 통일되지 않는다.

Q9. 주문승인(RESERVED→CONFIRMED/PRODUCING) 시점에 재고를 즉시 차감하는지, 아니면 출고(§5-6, "재고가 충분해진 CONFIRMED 주문") 시점에 재확인·차감하는지가 SPEC에 없다. 이걸 잘못 정하면: 승인 시 차감을 안 하면 동시에 여러 주문이 "재고충분"으로 오판되어 실제 재고보다 많은 주문이 CONFIRMED 처리되는 오버부킹이 발생한다.

Q10. 여러 주문이 동일 시료에 대해 동시에 승인 요청될 때의 순서/동시성 규칙이 SPEC에 없다. 이걸 잘못 정하면: 재고량 이상으로 여러 주문이 동시에 CONFIRMED로 전환되어, 실제 보유 재고보다 많은 수량이 출고 승인되는 정합성 오류가 생긴다.

Q11. 생산큐(FIFO)의 여러 항목이 하나의 생산라인에서 순차 생산되는지, 병렬 생산이 가능한지가 SPEC에 없다. 이걸 잘못 정하면: 총생산시간이 개별 항목 기준인지 라인 누적 기준인지에 따라 완료 시점(PRODUCING→CONFIRMED 전이 타이밍) 추정이 실제와 어긋나 "생산현황표기"·"대기주문확인" 화면의 예상 정보가 신뢰를 잃는다.

Q12. 주문수/재고수량/평균생산시간 등 수치 속성의 하한(0 이상)·자료형(정수/실수) 검증 규칙이 SPEC에 없다. 이걸 잘못 정하면: 음수 주문량이나 0 이하 평균생산시간이 그대로 등록되어도 시스템이 이를 받아들여, 이후 모든 계산식이 무의미한 값을 산출하게 된다.

Q13. 실생산량(ceil 결과, 수율 미달분 포함 가능성이 있는 생산 개수) 중 실제로 "재고"에 반영되는 수량이 실생산량 전체인지, 실생산량에 수율을 다시 곱한 "정상 시료 수"인지 SPEC에 없다("실생산량"이라는 이름 자체가 "생산 시도 횟수"인지 "정상품 개수"인지 모호함). 이걸 잘못 정하면: 재고수량 계산이 수율을 이중 반영하거나 전혀 반영하지 않아, 재고가 실제 정상품 수량과 어긋난다.

Q14. 시료검색(§5-1.3)의 "속성으로 검색" 대상 범위(이름만인지 시료ID/수율 등도 포함인지)가 SPEC에 없다. 이걸 잘못 정하면: 기능 검증(테스트) 시 팀마다 기대하는 검색 결과가 달라져 수용 기준이 성립하지 않는다.

Q15. 주문을 지칭하는 식별자(주문ID)가 "특정 주문"이라는 표현으로 §3·§5-3·§6에서 반복 사용되지만, 그 식별자 자체(존재 여부, 채번 방식)는 SPEC에 정의되어 있지 않다. 이걸 잘못 정하면: "특정 주문"을 지정하는 방법(식별자 체계)이 팀마다 달라져 영속성 스키마, 콘솔 명령 인자, 테스트 데이터 형식이 서로 호환되지 않는다.

Q16. 곱셈 기호 "×" 표기 지시(§4 하단 각주)가 정식 UI 섹션(§5, §6)에는 동일한 각주 형태로만 반복되고, 실제로 확정 반영되었다는 서술이 SPEC 어디에도 없다("OPEN ITEMS" 1·3번이 이미 이 상태를 인정). 이걸 잘못 정하면: UI 표기 일관성이 스펙 준수 심사 기준에서 빠져, "×" 적용 여부가 구현체마다 갈린다.
